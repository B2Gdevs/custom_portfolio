# Decisions

| Id | Status | Summary | Consequence |
| --- | --- | --- | --- |
| `BOOKS-DOCS-PLANNING-FIRST` | `accepted` | books docs expose planning pages before general implementation docs | section users hit active planning context first |
| `BOOKS-HOMEPAGE-READER-FIRST` | `superseded` | homepage reader was treated as the main reading surface, not a teaser | replaced by the dedicated reader route as the canonical runtime |
| `BOOKS-HOMEPAGE-READER-LAZY-MOUNT` | `superseded` | homepage reader mounted only when the user explicitly asked for it | replaced by the homepage launch-surface approach |
| `BOOKS-HOMEPAGE-READER-BYTE-LOAD` | `accepted` | embedded reader should load the `.epub` archive into memory before mounting `react-reader` | runtime behavior does not depend on browser MIME handling or URL-style EPUB fallback |
| `BOOKS-HOMEPAGE-LAUNCHPAD` | `accepted` | homepage is the launch surface for books and routes into `/apps/reader?book=<slug>`; legacy **`/books/**` pages removed** with **308** to `/apps/reader` (static EPUB files stay under `/books/<slug>/book.epub`) | the reader app route is the canonical live reading runtime |
| `BOOKS-DEDICATED-READER-BOOK-SWITCHER` | `accepted` | active reading sessions should expose book switching in the read-page header | readers can move between available books without leaving the reader route |
| `BOOKS-DEDICATED-READER-SPREADS` | `accepted` | wide screens should use paginated two-page spreads when the viewport allows it | the reading surface can feel like an opened book as long as each emitted page shell stays compact enough to fit |
| `BOOKS-DEDICATED-READER-TOC-STYLING` | `accepted` | TOC chrome and in-book TOC typography should feel designed, not default | navigation surfaces should look intentional even before content-specific polish |
| `BOOKS-DEDICATED-READER-STAGE-CLEARANCE` | `accepted` | the spread needs extra left clearance from the TOC toggle chrome in reader mode | reading pages should not feel pinned under the hamburger button |
| `BOOKS-DEDICATED-READER-BOTTOM-NAV` | `accepted` | dedicated reader navigation should live in a compact bottom strip with section title and page movement controls | reading navigation stays accessible without adding louder chrome to the header or page edges |
| `BOOKS-DEDICATED-READER-CUSTOM-TOC` | `accepted` | the dedicated reader should use a custom collapsed contents panel instead of the built-in `react-reader` TOC chrome | the menu stays fully hidden until opened and no hidden TOC text bleeds into the reading stage |
| `BOOKS-LEAD-IMAGE-PAGES-CAN-SPLIT` | `superseded` | on wider spreads, a lead-image story page can lay out with the figure on the left and the opening text beside it | replaced by the stricter single frame-shape rule for all illustrated pages |
| `BOOKS-IMAGE-FRAME-SHAPE-IS-FIXED` | `accepted` | every illustrated page should use the same fixed frame shape as the placeholder slot, even if the image needs to crop | art presentation stays visually consistent and the page shell does not change shape based on source aspect ratio |
| `BOOKS-MANUSCRIPT-STRUCTURE-FIRST` | `accepted` | books should be authored as structured manuscripts with chapter metadata and content pages, not as filename-driven display titles | the builder must derive paperback-style output from formal source structure instead of exposing raw page filenames |
| `BOOKS-STORY-PAGES-STAY-IN-SPINE` | `accepted` | story pages can be hidden from the TOC but must stay in the EPUB spine | TOC cleanup must never remove the actual reading order from the built book |
| `BOOKS-EMIT-PAGE-SHELLS` | `accepted` | EPUB generation should emit custom page wrappers with lead-image placement and bottom folios | the dedicated reader can treat source files like intentional printed pages instead of raw chapter fragments |
| `BOOKS-LEAD-IMAGES-LIFT-OUT-OF-PARAGRAPHS` | `superseded` | builder should treat a leading inline image in the first paragraph as the page lead figure | replaced by the simpler fixed illustration slot rule for each authored page |
| `BOOKS-DEDICATED-READER-SINGLE-PAGE` | `superseded` | the dedicated reader should always render one visible page at a time, even on wide screens | replaced by the compact-page-shell spread approach for wide screens |
| `BOOKS-PAGE-IMAGE-SLOT-REQUIRED` | `accepted` | every emitted page should reserve a fixed illustration slot before the text body, falling back to placeholder art when no page image exists | page rhythm stays consistent across the manuscript and missing art does not collapse the layout |
| `BOOKS-MISSING-IMAGE-REFS-DO-NOT-SHIP` | `accepted` | EPUB generation must strip book image refs when the source illustration file is missing | reader editions prefer missing art over broken asset errors |
| `BOOKS-REPO-SOURCE-OF-TRUTH` | `accepted` | authored books under `books/<slug>/` remain the publishing source of truth | updates must follow the repo content pipeline, not manual copies |
| `BOOKS-ANNOTATIONS-TIERED-STORAGE` | `accepted` | highlights/comments live in IndexedDB keyed by book id + EPUB content hash; optional embedded JSON in EPUB zip under `META-INF/portfolio-annotations.json` | third-party readers keep working; only aware tools read the extension |
| `BOOKS-READER-PLANNING-READONLY` | `accepted` | planning surfaces in the reader are read-only for visitors; any future in-browser editing of repo planning requires explicit gating | public session cannot mutate manuscript or `.planning` |
| `BOOKS-READER-LIBRARY-SHELL` | `accepted` | the **canonical reading experience** is an **app shell** at `/apps/reader` that includes a **library / overview** surface (not only a single open book); the old **`/books` listing routes are removed** (redirect to reader); EPUB **files** remain at **`/books/<slug>/book.epub`** under `public/books/` | implement **`books-reader-03`**; see [Route conventions](/docs/documentation/route-conventions) |
| `BOOKS-READER-RUNTIME-REPUB-PACKAGE` | `accepted` | **Reader shell + workspace** should be **exported from `@portfolio/repub-builder`** (or a dedicated export subpath such as `./next` / `./runtime`), not a separate ad hoc package name; optional **Next route-handler** helpers for third-party hosts, **CopilotKit-style** | **`build-books` / CLI** and **in-browser reader** share one EPUB/repub vertical; evolve package `exports` with an ADR when the first runtime PR lands |
| `BOOKS-AI-COPILOTKIT-V1` | `superseded` | the first public chat shipped on CopilotKit + OpenAI as a read-only assistant | replaced by the assistant-ui shell once citation-backed RAG and bespoke chat chrome became first-class needs |
| `BOOKS-AI-ASSISTANT-UI-SHELL` | `accepted` | the public site chat shell runs on assistant-ui with a local runtime, a thin `/api/chat` backend route, Claude-style modal chrome, and development-only devtools access | the frontend owns the chat UI/state while the backend stays responsible for RAG grounding and OpenAI calls |
| `BOOKS-AI-HARD-CUTOVER` | `accepted` | once the assistant-ui shell shipped, the public chat stack should remove the old CopilotKit route and packages instead of keeping a compatibility layer | there is exactly one public chat path to maintain: assistant-ui + `/api/chat` |
| `BOOKS-AI-RAG-PAYLOAD-SQLITE-VEC` | `accepted` | planned RAG backend uses Payload CMS over SQLite for sources/chunks/ingest records and `sqlite-vec` for embeddings + KNN search | repo-authored content remains canonical; the DB stores indexed retrieval copies and vectors |
| `BOOKS-AI-RAG-SIDECAR-INDEX-DB` | `accepted` | Payload keeps canonical RAG records in its own SQLite file, while FTS and `sqlite-vec` live in a sidecar RAG SQLite file | Payload schema introspection stays stable, while the vector/FTS index can load `sqlite-vec` without coupling that extension to the Payload connection |
| `BOOKS-AI-RAG-CITATIONS-FIRST` | `accepted` | AI answers should be able to expose retrieved chunks with source labels, URLs, and anchors | retrieval payloads must preserve provenance metadata, not just raw chunk text |
| `BOOKS-AI-RAG-REPO-INGEST` | `accepted` | "know more about me" comes from chunking/indexing repo and other owner-managed source material, not a hand-curated static about-me prompt file | ingest/chunking quality matters more than adding a bespoke author bio document |
| `BOOKS-AI-RAG-SOURCE-SCOPE` | `accepted` | ingest `content/docs`, `content/projects`, `content/blog`, and books/world-building material; exclude the raw reader-facing manuscript by default | public chat should know the world, characters, and work without treating draft manuscript pages as the main retrieval corpus |
| `BOOKS-AI-RAG-MAGICBORN-SECTION` | `accepted` | worldbuilding docs live in a dedicated `magicborn` docs section and act as the primary public lore corpus for chat retrieval | world/lore writing has a clear home in the docs IA and does not get conflated with manuscript chapters |
| `BOOKS-AI-RAG-INGEST-SCRIPT-FIRST` | `accepted` | support both manual/scripted ingest and an owner-only admin trigger, but treat the script path as canonical | indexing stays operable without depending on the admin UI, and admin remains convenience rather than the only workflow |
| `BOOKS-AI-RAG-SOURCE-PER-PAGE` | `accepted` | create one source record per public page instead of grouping multiple pages into one family document | citations, checksums, and incremental ingest stay simpler and more stable |
| `BOOKS-AI-RAG-EMBEDDING-TARGET` | `accepted` | default embedding target is OpenAI `text-embedding-3-small` at `1536` dimensions with cosine distance | quality stays high for a mixed but modest corpus without prematurely optimizing for storage or latency |
| `BOOKS-AI-RAG-PAYLOAD-OWNS-CHUNKS` | `accepted` | Payload stores canonical chunk text and citation metadata; the sidecar FTS/`sqlite-vec` DB stores vectors keyed by `chunkId` plus only small filter fields | retrieval can use vector search without duplicating large text blobs in the vec table or forcing vec tables into the Payload DB |
| `BOOKS-AI-RAG-CHUNK-BODY-STORED` | `accepted` | each chunk record stores its normalized chunk body directly | retrieval, citations, and debugging do not depend on reconstructing text from offsets |
| `BOOKS-AI-RAG-LAST-COMPLETED-RUN` | `accepted` | chat should only read from the last fully completed ingest run, never a half-rebuilt corpus | reindexing stays operationally simple and avoids mixed old/new retrieval results |
| `BOOKS-AI-RAG-SOFT-DELETE-AUDIT` | `accepted` | removed sources may remain soft-deleted in Payload for audit, but their active FTS/vector rows should disappear from retrieval as soon as the replacement run commits | admin can retain history without leaking retired content back into chat |
| `BOOKS-AI-RAG-STABLE-SOURCE-IDS` | `accepted` | source ids stay stable across runs, while chunk ids may be regenerated per run | citations and source links can stay stable even when chunk boundaries shift after edits |
| `BOOKS-AI-RAG-HYBRID-RERANK` | `accepted` | retrieval should add a lightweight rerank/filter pass after vector search instead of relying on pure top-k | metadata checks plus a small lexical/FTS pass should improve exact-name and title-sensitive queries |
| `BOOKS-AI-RAG-RERANK-BM25` | `accepted` | the lexical rerank stage should use SQLite FTS5 BM25 over the vector-candidate chunk set, not token overlap only | exact names, titles, and phrase-sensitive queries should recover better without abandoning semantic recall |
| `BOOKS-AI-RAG-FTS-TEXT-PLUS-METADATA` | `accepted` | FTS should index chunk text plus selected metadata like titles, headings, aliases, and section labels | lexical rerank can recover exact-title and exact-name queries more reliably |
| `BOOKS-AI-RAG-ANCHORED-CITATIONS` | `accepted` | citations should preserve heading or section anchors when available | source cards can link users to the precise part of a page, not just the page root |
| `BOOKS-AI-RAG-INCREMENTAL-CHECKSUM-INGEST` | `accepted` | ingest should rebuild only changed sources when checksums differ | v1 stays fast enough to rerun often without full-corpus rebuilds |
| `BOOKS-AI-RAG-AUDIT-HIDDEN-BY-DEFAULT` | `accepted` | soft-deleted sources and old ingest runs should be hidden in normal admin views by default | the owner UI stays clean while retaining auditability |
| `BOOKS-AI-RAG-CITATION-PANEL` | `accepted` | chat renders source chunks inside a collapsible Sources panel by default | answers stay cleaner while still exposing provenance and snippets on demand |
| `BOOKS-AI-ADMIN-BASIC-AUTH` | `accepted` | Payload admin and ingest/reindex controls use basic auth in production | public chat can stay open/read-only without shipping an unauthenticated admin surface |
| `BOOKS-AI-SCOPED-LOGGING` | `accepted` | client and server AI flows should use one shared logger with env-driven levels and scope filters instead of permanent ad hoc console statements | chat/RAG debugging can become verbose in development and targeted sessions, then be quiet again without deleting instrumentation |
| `BOOKS-AI-STACK-NEXT` | `accepted` | assistant-ui is the active public shell; follow-on evaluations now narrow to Tool UI for schema-first tool rendering and MCP-oriented surfaces only if the assistant becomes more tool-heavy later | avoid parallel public chat stacks; add new UI/runtime layers only when a concrete feature forces them |
