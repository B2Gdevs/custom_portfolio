# Repo Planner - Decisions

## `REPOPLAN-SHELL-01` -- Cockpit shell lives in the imported package

| Field | Value |
| --- | --- |
| **Decision** | Project switcher, upload pack, export workspace, init-template downloads, and the dashboard chrome around `PlanningCockpit` are owned by the `repo-planner` package and imported into the portfolio. The host app does not keep a long-term parallel shell once the package exports a single host-mountable shell. |
| **Rationale** | One UX surface, one place to fix bugs; portfolio only supplies routes, API handlers, env, and host-specific props. |
| **Consequences** | `repo-planner-integration-03` includes upstream or transitional extraction of that shell; the portfolio page becomes a thin import plus layout wrapper. |

## `REPOPLAN-EMBED-01` -- Public embed is read-only on the server

| Field | Value |
| --- | --- |
| **Decision** | Embedded cockpit at `/apps/repo-planner` stays read-only on the server by default. GET routes may read committed `.planning/`; write routes return `403` unless explicitly enabled by env flag. No portfolio-hosted planning LLM is assumed. |
| **Rationale** | Same spirit as the EPUB reader: browser-local and export-to-git, not casual server mutation. |
| **Consequences** | Operators use `pnpm planning` or standalone RepoPlanner for write-capable workflows. |

## `REPOPLAN-REPORTS-01` -- CLI telemetry lives outside `.planning/`

| Field | Value |
| --- | --- |
| **Decision** | `REPOPLANNER_REPORTS_DIR` defaults to `.planning-reports/` at repo root so usage logs and generated reports do not recreate `.planning/reports/` after minimal init. |
| **Rationale** | Bare `.planning/` stays XML plus config only; reports are local operator artifacts. |

## `REPOPLAN-CLI-01` -- Two CLIs, `dlx`-first publish story

| Field | Value |
| --- | --- |
| **Decision** | Root `pnpm planning ...` remains the monorepo operator wrapper, but RepoPlanner now also exposes a formal `repo-planner` bin from the package itself. Local/file installs should use `pnpm exec repo-planner ...`; published `pnpm dlx repo-planner ...` and global installs should reuse the same entrypoint when the package is published. |
| **Work ids** | `documentation-site-07-01`, `repo-planner-integration-01-13` |

## `REPOPLAN-PUBLISH-01` -- Source-first public package with semver gates

| Field | Value |
| --- | --- |
| **Decision** | RepoPlanner publishes as `repo-planner` with a source-first package contract: runtime source files, CLI scripts, exported subpaths, and `.planning/` bootstrap assets ship in the tarball; React/Next/UI runtime packages remain peer dependencies. |
| **Rationale** | The package is primarily an embeddable host surface plus CLI, so consumers need the source entrypoints and bootstrap assets more than a separate build artifact. Keeping React/Next in peers prevents the package from pinning the host runtime. |
| **Consequences** | Use semver intentionally: patch for docs/package/non-breaking fixes, minor for additive exports or commands, major for export-path or CLI-contract breaks. Verify with `pnpm pack --dry-run` before calling the publish story done. |

## `REPOPLAN-ROUTES-01` -- Hand-maintained Next API routes

| Field | Value |
| --- | --- |
| **Decision** | RepoPlanner's route installer now targets package subpath imports and monorepo app roots like `apps/portfolio`, but it stays read-only by default: GET routes only unless the operator explicitly opts into `planning-cli/run` or `planning-edits/apply`. This repo may continue to keep explicit `app/api/planning-*` routes where host-local behavior is clearer than generated stubs. |

## `REPOPLAN-UPLOAD-01` -- Preview uploads stay ephemeral and read-only

| Field | Value |
| --- | --- |
| **Decision** | Pack mode distinguishes between editable local imports and preview uploads. `Import` creates a writable browser-local pack. `Preview upload` accepts pack JSON or `.planning` zip input, opens it read-only, does not persist it into local storage, and does not expose a production write path. |
| **Rationale** | Operators sometimes need to inspect a planning bundle quickly without mutating workspace state or implying a live repo connection. |
| **Consequences** | Tracked by `repo-planner-integration-01-06`; Pack copy and sidebar labels must make the ephemeral/read-only distinction explicit. |

## `REPOPLAN-PACK-MODAL-01` -- Planning pack copy

| Field | Value |
| --- | --- |
| **Decision** | Planning pack modal copy should state that packs align with [RepoPlanner](https://github.com/MagicbornStudios/RepoPlanner) conventions and link upstream. |

## `REPOPLAN-WORKFLOW-01` -- RepoPlanner accelerates the loop; it does not replace it

| Field | Value |
| --- | --- |
| **Decision** | RepoPlanner should summarize and scaffold the repo's planning loop: Global docs stay the human-first entrypoint, while the CLI and cockpit help agents pick phases faster, create kickoff records, surface missing definition-of-done fields, and refuse executable-behavior closeout without tests. |
| **Rationale** | The tool should reduce rereading and bookkeeping while still preserving the roadmap/state/task-registry/decisions model. |
| **Consequences** | Planned work includes workflow-aware snapshot, kickoff scaffolding, done gates, ownership guidance, and cockpit visibility for missing tests or missing DoD data in `repo-planner-integration-05`. |

## `REPOPLAN-MODES-01` -- Pack mode vs Live mode in the embed

| Field | Value |
| --- | --- |
| **Decision** | The portfolio embed exposes two explicit data modes: Pack (built-in gallery plus user uploads) and Live (server reads the mounted repo's `.planning/` via GET APIs). UI copy, empty states, and sidebar labels must not imply a live monorepo connection when the user is in Pack-only mode. |
| **Rationale** | Browser code cannot read arbitrary disk paths; only Live mode justifies "this repository" semantics. |
| **Consequences** | Tracked as `repo-planner-integration-03-08` and `REPOPLAN-REQ-10`. |

## `REPOPLAN-SNAPSHOT-01` -- Snapshot is a steering surface, not a document reader

| Field | Value |
| --- | --- |
| **Decision** | Snapshot output should default to structured steering data: ranked active/planned phases only, why-now reasons, blocker warnings, ownership targets, sprint status, completion percent, effort labels, stale signals, missing-test or missing-DoD warnings, and unresolved phase open questions. When a phase crosses the `needs discussion` threshold, the recommended action should shift from implementation toward discussion. Snapshot should not dump whole planning pages into the main surface. |
| **Rationale** | Agents need the next good action faster than they need prose recall. The file tree already preserves the document model; snapshot should reduce scanning cost. |
| **Consequences** | Tracked by `REPOPLAN-REQ-13`, `repo-planner-integration-05-01`, and `repo-planner-integration-06-04`. |

## `REPOPLAN-AGENTS-01` -- Show AGENTS as a compact rules reminder

| Field | Value |
| --- | --- |
| **Decision** | RepoPlanner should show a compact `AGENTS.md` reminder card in snapshot and cockpit views: read order, **kickoff when vague / stale / estimated hours over policy max** (not for every phase — **`REPOPLAN-KICKOFF-01`**), tests-required-for-done, and a deep link to the full file. Full AGENTS prose stays available as a linked source, not as the default pane content. |
| **Rationale** | Operators need the loop rules visible, but the full document is too dense to inline in every workflow surface. |
| **Consequences** | Tracked by `REPOPLAN-REQ-14` and `repo-planner-integration-05-07`. |

## `REPOPLAN-UI-01` -- File tree for navigation, structured panes for work

| Field | Value |
| --- | --- |
| **Decision** | Keep the file tree as the primary navigation model, but render known planning files as structured cards, lists, tables, badges, forms, sprint views, and percentages in the main pane. Raw markdown or XML stays out of the normal RepoPlanner product surface. |
| **Rationale** | Planning files are good storage and traceability units, but poor day-to-day steering surfaces when rendered as full documents. |
| **Consequences** | Tracked by `REPOPLAN-REQ-15`, `REPOPLAN-REQ-16`, and phase `repo-planner-integration-06`. |

## `REPOPLAN-SCORING-01` -- Weighted recommendation formula stays explicit

| Field | Value |
| --- | --- |
| **Decision** | Use an explicit weighted score to rank candidate phases. Baseline recommendation score = `strategic-importance + unblock-value + urgency + momentum + test-followup-value + recency-boost - shrinking-global-penalty - effort-penalty - blocked-penalty - open-question-entropy penalty`. Missing tests are a positive pickup signal for open executable work, not a recommendation penalty. Created/updated timestamps act as tie-breakers or small boosts only. Each open question contributes equally to entropy, and `needs discussion` lowers implementation recommendation rather than acting as badge-only metadata. The simplified action set stays `Implement now`, `Discuss first`, and `Unblock dependency`; when both blockers and unresolved-question entropy are present, `Discuss first` outranks `Unblock dependency` so the recommendation reflects planning risk instead of pretending the only next step is code. |
| **Rationale** | CLI and cockpit must recommend the same work for the same data, and operators need to understand why one phase outranks another. High question count means high planning entropy, which is a poor random implementation pickup even when the phase is strategically important. |
| **Consequences** | Tracked by `repo-planner-integration-05-01` and `repo-planner-integration-05-06`. |

## `REPOPLAN-KICKOFF-01` -- Effort threshold uses estimated hours

| Field | Value |
| --- | --- |
| **Decision** | **Above effort threshold** means the phase’s **estimated hours** exceed a **host-configured maximum** (policy). **Estimated hours** come from an explicit field on the phase (roadmap / registry row) or, when absent, from **summed task estimates** under that phase where the format supports it. Vague and stale triggers are unchanged. |
| **Rationale** | Hours match scheduling and session planning better than raw task counts. |
| **Consequences** | `repo-planner-integration-05-02`, `repo-planner-integration-05-06`, and schema work document the field names, fallback sum rule, and default cutoff. |

## `REPOPLAN-POLICY-01` -- Portfolio workflow rules are host policy, not invisible upstream defaults

| Field | Value |
| --- | --- |
| **Decision** | RepoPlanner should support host-configured workflow policy. This portfolio can require Global-first read order, **kickoff when a phase is vague, stale, or above estimated-hours threshold** (see **`REPOPLAN-KICKOFF-01`**), and tests-required-for-done, while other projects may choose different rules. Upstream defaults should stay generic and discoverable. |
| **Rationale** | The product needs to stay reusable outside this repo even while this repo drives its design. |
| **Consequences** | Tracked by `REPOPLAN-REQ-18` and `repo-planner-integration-06-06`. |

## `REPOPLAN-FORMS-01` -- Pack editing autosaves locally; IDs are immutable in normal flows

| Field | Value |
| --- | --- |
| **Decision** | RepoPlanner form editing uses debounced client-side autosave with visible save-state feedback. Live mode remains read-only. IDs are immutable in normal editing once created; any rename is an advanced migration action that rewrites references and is unavailable in Live mode. |
| **Rationale** | Operators want low-friction editing, but ids are reference keys and Live mode must stay honest about mutability. |
| **Consequences** | Tracked by `REPOPLAN-REQ-21`, `REPOPLAN-REQ-22`, `repo-planner-integration-06-02`, and `repo-planner-integration-06-07`. |

## `REPOPLAN-QUESTIONS-01` -- Phase open questions stay visible even when non-blocking

| Field | Value |
| --- | --- |
| **Decision** | Every phase may carry visible open questions surfaced by agents during planning or execution. Questions stay phase-scoped, use a single text field, stay in created order, do not block phase start by default, and use only two states: `open` and `answered` (decision recorded). In the UI, answered questions should live in a collapsed history area rather than disappearing. Two or more open questions should trigger a `needs discussion` signal in the planning surface. When a question is marked answered, RepoPlanner may create a new decision entry in the same non-AI inline composer flow and then backfill the linked decision reference. |
| **Rationale** | Agent uncertainty should be captured in planning memory instead of disappearing into chat logs, while answered questions should stay reviewable without crowding the main surface. A small number of questions is normal; multiple open questions indicate higher planning entropy and a better candidate for human discussion than random implementation pickup. The question workflow is plain UI support, not an in-product AI planning feature. |
| **Consequences** | Tracked by `REPOPLAN-REQ-23`, `repo-planner-integration-05-05`, `repo-planner-integration-05-06`, and `repo-planner-integration-06-08`. |

## `REPOPLAN-ROOTS-01` -- Config-declared planning directories, not implicit doc-tree scans

| Field | Value |
| --- | --- |
| **Decision** | CLI snapshot, scoring, and bundle builders resolve planning inputs **only** from **`planning-config.toml`** (or successor) declarations: one or more **workspace roots**, optional **descendant discovery** under a parent, and a **planning directory basename** per root (default **`.planning`**, with support for names like **`planning`**). The conventional monorepo case remains a single repo-root **`.planning/`** as the global machine loop; **global** may be remapped to another path. The tool does **not** crawl arbitrary subtrees (e.g. all MDX under `content/docs`) unless a root + discovery rule explicitly includes them. |
| **Rationale** | Predictable performance, correct multi-root monorepos, and no accidental coupling to one site’s docs layout. |
| **Consequences** | Tracked by `REPOPLAN-REQ-24`, `repo-planner-integration-03-03`, `repo-planner-integration-05-01`, and future `planning-config.toml` schema work. Live GET routes should use the same resolver as snapshot/scoring instead of hard-coding `.planning` in each handler. |
