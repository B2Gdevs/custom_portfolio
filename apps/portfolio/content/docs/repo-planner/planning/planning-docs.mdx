## Deprecation and upstream (MagicbornStudios/RepoPlanner)

**Product stance:** RepoPlanner is **deprecated** in favor of **GAD** (`gad` CLI, root `.planning/`, `gad-config.toml`). See **[Global decision 06-01](/docs/global/planning/decisions)** and `.planning/REPOPLANNER-TO-GAD-MIGRATION-GAPS.md` (monorepo root). Phase **06** task **06-03** tracks removing the `repo-planner` package from this portfolio; this docs section stays for **host contracts** and migration history until that work finishes.

**Upstream reference site:** `RepoPlanner`’s **`apps/landing`** deploys to Vercel (static Next.js: marketing copy, cockpit demo, built-in `builtin-packs.json`). It is **not** ongoing product work. **Narrow exception:** **build-only** fixes so `npm run build` stays green — **direct** `npm` dependencies for transitive peers the embed pulls in, and **`link-landing-deps-to-repo-root.mjs`** (`postinstall` + `prebuild`) symlinks landing `node_modules` into the repo root so TypeScript can resolve `react`, `next`, and `@types/*` when typechecking `components/**` (npm hoists to `apps/landing` only; see task **06-04** in root `.planning/TASK-REGISTRY.xml`). When in doubt, **GAD** and **portfolio** docs are canonical; upstream is a **read-only** reference.

## Workflow stance

- **Roadmap, state, task registry, and decisions** are the living records.
- Any one-off phase plan should live under a section planning folder such as `planning/plans/<phase-id>/`.
- `IMPLEMENTATION_PLAN.md` should not exist in the minimal root tree; if an older clone still has it, migrate the useful content into roadmap/state/task-registry/decisions or section plans and delete the file.
- RepoPlanner should align with the same structure instead of encouraging parallel "plan vs task registry" workflows.

## Host embed on this site

**Intent:** Browser edits stay **local / export → git**; the server does not persist casual cockpit mutations. Full detail: [Decisions](/docs/repo-planner/planning/decisions) **`REPOPLAN-EMBED-01`**, **`REPOPLAN-SHELL-01`**, **`REPOPLAN-MODES-01`**.

**Modes:** **Pack** (gallery + uploads; client-side bundle where possible) vs **Live** (GET APIs reading this repo’s **`.planning/`**). That mode split, toolbar shell, init download flow, and contrast pass are now shipped in phase `repo-planner-integration-03`; workflow support is now shipped in phase `repo-planner-integration-05`, and follow-on work moves into structured editing (`06`).

| Surface | Default behavior |
| --- | --- |
| **GET** planning APIs | Read committed planning files through the shared config/root resolver (`planning-config.toml` roots; repo-root **`.planning/`** fallback only when config does not declare roots). Live bundle JSON is built directly from XML / planning files rather than CLI spawn per poll |
| **`POST /api/planning-edits/apply`**, **`POST /api/planning-cli/run`** | **403** unless **`PLANNING_COCKPIT_ALLOW_SERVER_WRITES=1`** |
| **Planning chat / in-site LLM** | **Out of scope** -- **`POST /api/ai/planning-chat`** returns **501** unless you wire a backend |
| **Workspace chrome** (project switcher, pack upload, export, init downloads) | Owned by the **`repo-planner`** package and imported; portfolio mounts the shell and supplies routes/env (**`REPOPLAN-SHELL-01`**). Live and Pack now share the same vendored **`PlanningCockpit`** body through the shared data-source contract rather than keeping a host-only fallback pane. |

## Workflow support shipped

- Snapshot ranking, current-sprint context, and the compact `AGENTS.md` reminder now come from one shared workflow engine across CLI, Live, and Pack.
- `workflow kickoff`, `workflow done-gate`, and `workflow ownership` now expose the same phase steering data the cockpit uses.
- Kickoff output now includes a reusable block template, with a committed example at `planning/plans/repo-planner-integration-05/PLAN.mdx`.
- Executable work cannot be marked done through the CLI without tests on the verification path.
- The cockpit dashboard now surfaces workflow cards for recommendation, kickoff state, done-gate warnings, ownership guidance, open questions, and collapsed answered-question history.
- The Pack overview tab now surfaces workflow summary cards and host-policy badges instead of only raw file KPIs.
- The inspector now renders structured requirement, roadmap, state, decisions, kickoff, task-registry, and `-PLAN.xml` phase-plan views with host-policy badges and locked-id guidance rather than defaulting to raw markdown/XML.
- Phase-question answers now use a non-AI inline composer that can create a decision record, backfill the linked reference, and keep answered history collapsed in the same inspector surface.
- Advanced id migration now exists only for writable local packs and rewrites stable string-id references across pack files; Live mode and short numeric/root ids stay locked.

### Production notes

**GET** handlers now read planning files directly through the shared resolver instead of spawning the CLI for each poll. Confirm the configured planning root(s) and reports dir are present in the deploy artifact if you want Live data. With default **read-only** POST policy, HTTP does not write planning files.
