**CLI dependencies** (`commander`, `fast-xml-parser`, `@iarna/toml`, `ejs`) are installed at the **workspace root** so `node` can resolve them when executing the submodule script.

## Project root resolution

By default the CLI uses **`process.cwd()`**. Set **`REPOPLANNER_PROJECT_ROOT`** (path relative to cwd) or pass **`--root <path>`** when running from another directory. Set **`REPOPLANNER_REPORTS_DIR`** (relative to project root or absolute) to control where **`usage.jsonl`** / report **`latest.md`** go ‚Äî this monorepo defaults it to **`.planning-reports/`** via **`scripts/run-planning-cli.mjs`** and **`next.config.ts`**.

## Product boundary (portfolio embed)

**Embedded cockpit intent:** edits in the browser are **local to the UI**; operators **export** and **commit from git** (same spirit as the EPUB reader ‚Äî nothing long-lived on the server from casual browsing). The portfolio does **not** rely on HTTP endpoints to persist cockpit edits to disk.

| Surface | Audience | Behavior |
| --- | --- | --- |
| **Embedded cockpit** ([`/docs/apps/repo-planner`](/docs/apps/repo-planner)) | Anyone who can open the page | **UI + read-only server data** by default: **GET** routes can run **`simulate loop --json`** (read committed `.planning/` under `REPOPLANNER_PROJECT_ROOT`). **No portfolio-hosted AI** for planning; **Chat** tab stays inert unless you add your own backend. |
| **Server writes / in-browser CLI** | Disabled by default | **`POST /api/planning-edits/apply`** and **`POST /api/planning-cli/run`** return **403** unless **`PLANNING_COCKPIT_ALLOW_SERVER_WRITES=1`**. See **What ‚Äúapply‚Äù and CLI POST are** below. |
| **RepoPlanner standalone** (clone upstream + `web` / desktop) | Operators | Full upstream behavior (AI + Codex + CLI) when **you** run that app locally ‚Äî **not** the portfolio embed. |
| **`pnpm planning` / `pnpm planning:ui`** (repo root) | Operators | Real CLI on your machine; **usage + generated reports** go to **`.planning-reports/`** (not `.planning/reports/`) when using root **`pnpm`** scripts. |
| **Portfolio docs + planning pack** | Visitors | Read-only MDX + downloadable `.md` exports. |
| **Future: upload `.planning` zip** | Visitors | Parse/view in memory ‚Äî **ephemeral**, no git mutation on the host. |

### What ‚Äúapply‚Äù and CLI POST are (upstream vs portfolio)

| Mechanism | What it does in RepoPlanner upstream | Portfolio embed default |
| --- | --- | --- |
| **`POST /api/planning-edits/apply`** | Persists **AI chat‚Äìproposed** edits by **writing files under `.planning/`** on the server (`mkdir` / `writeFile`). | **403** ‚Äî not part of the export-only workflow; enable only with **`PLANNING_COCKPIT_ALLOW_SERVER_WRITES=1`**. |
| **`POST /api/planning-cli/run`** | Spawns **`loop-cli.mjs`** (can run `report generate`, `snapshot`, etc., which may **write** reports or telemetry). | **403** by default ‚Äî same flag to enable for operator sandboxes. |
| **`GET /api/planning-state`** | Runs **`simulate loop --json`** to **read** current loop bundle. | **On** (read-only path to committed XML). |

**Roadmap (upstream RepoPlanner):** shift the cockpit so **all** editing state lives **client-side** and **export** is the only artifact path ‚Äî then `apply` / CLI POST become unnecessary for embeds. Tracked as portfolio alignment work in the [task registry](/docs/repo-planner/task-registry).

## Bare `.planning` (default) vs full upstream init

**Contract:** **`.planning/`** is the **root planning section** (XML + **`planning-config.toml`** + **`.planning/AGENTS.md`**) ‚Äî same *workflow* as **`content/docs/<section>/`**, different *format*. **Human** monorepo narrative + cross-cutting queue: **[documentation/requirements](/docs/documentation/requirements)** + **[documentation/state](/docs/documentation/state)**; repo-root **`REQUIREMENTS.md`** is a **stub pointer** only. **Phases and tasks** live in **`ROADMAP.xml`** / **`TASK-REGISTRY.xml`** / **`STATE.xml`**. There is **no** `IMPLEMENTATION_PLAN.md`.

### What went wrong in the first `--minimal` pass

An earlier **`--minimal`** implementation still wrote **`REQUIREMENTS.md`** / **`IMPLEMENTATION_PLAN.md`** *inside* **`.planning/`**, which broke the ‚Äúbare tree‚Äù expectation. Separately, upstream **`loop-cli.mjs`** always appends **`usage.jsonl`** under **`getReportsDir()`** (default **`.planning/reports/`**), so **`pnpm planning snapshot`** recreated **`reports/`** even after a ‚Äúno reports‚Äù init. **Fixes in this repo:**

| Change | Purpose |
| --- | --- |
| **`runPlanningInitMinimal`** | On **`--force`**: copy narrative `.md` from **`.planning/`** ‚Üí repo root if root files are missing, then **delete** **`.planning/reports/`**, **`.planning/REQUIREMENTS.md`**, **`.planning/IMPLEMENTATION_PLAN.md`**. |
| **`REPOPLANNER_REPORTS_DIR`** | **`getReportsDir()`** in **`loop-cli.mjs`** uses this when set (absolute or relative to project root). |
| **`scripts/run-planning-cli.mjs`** | Defaults **`REPOPLANNER_REPORTS_DIR=.planning-reports`** for all **`pnpm planning*`** scripts. |
| **`next.config.ts`** + **`getRepoPlannerChildEnv()`** | Next **`spawn`** of the CLI gets the same reports path so the embed does not write under **`.planning/reports/`**. |

**Default `pnpm planning:init`** creates:

| Under **`.planning/`** | At **repo root** | Omitted (use **`pnpm planning:init:full`**) |
| --- | --- | --- |
| `planning-config.toml`, `STATE.xml`, `TASK-REGISTRY.xml`, `ROADMAP.xml`, `DECISIONS.xml`, `ERRORS-AND-ATTEMPTS.xml`, `REQUIREMENTS.xml` (stub ‚Üí **documentation** requirements + state paths), **`AGENTS.md`** (XML-layer playbook) | **`REQUIREMENTS.md`** at repo root (stub only if created by bootstrap ‚Äî **not** canonical prose) | `.planning/templates/`, `.planning/phases/`, `.planning/README.md` |

**Local-only:** **`.planning-reports/`** (gitignored) ‚Äî **`usage.jsonl`**, **`latest.md`**, etc.

**Archive:** **`.planning-archive/2026-03-29-pre-barebones/`** ‚Äî migrate prose into **`documentation/requirements.mdx`**, section registries, and **`.planning/*.xml`**.

## How API routes get installed (no CopilotKit-style single runtime)

Next.js **App Router** does not offer one **runtime registration** call that mounts every handler (unlike some adapter SDKs). RepoPlanner upstream ships **`scripts/install-routes.mjs`**, which **generates** thin `route.ts` files that re-export **`GET`/`POST`** from `vendor/repo-planner/api/...`. That script‚Äôs default **`--app-dir`** and relative `import` depth target a different layout than **`apps/portfolio`**, so this repo keeps **explicit** files under **`apps/portfolio/app/api/`** and shared helpers in **`lib/repo-planner/`**.

**Future options:** (1) extend **`install-routes.mjs`** upstream for **`--app-dir=apps/portfolio`** and correct relative imports; (2) add a **single catch-all** `app/api/repo-planner/[...path]/route.ts` that delegates to package handlers (more coupling, trickier types); (3) keep hand-maintained routes when the set is small and policy flags differ (e.g. read-only embed).

## Standalone cockpit (operators)

1. Clone [MagicbornStudios/RepoPlanner](https://github.com/MagicbornStudios/RepoPlanner) (or use a release / desktop binary per upstream README).
2. Install dependencies for the **web** app when present: e.g. `pnpm --dir web install` (exact path from upstream).
3. Point the app at **this** repository: `REPOPLANNER_PROJECT_ROOT=/absolute/path/to/custom_portfolio` then start the dev server from the RepoPlanner repo.
4. Edit in the cockpit, **export**, and commit from git‚Äîsame export-driven workflow as the in-site reader (no production write-back).

## Planning files: what to use when

This repo **does not** commit `.planning/templates/` (minimal init). Upstream **templates** live only inside **`vendor/repo-planner/.planning/templates`** for **`planning:init:full`**. **Live** files are what agents read; **`pnpm planning snapshot`** is **optional** for a merged CLI bundle.

| File | Role |
| --- | --- |
| **`ROADMAP.xml`** | **Product / theme timeline** in RepoPlanner terms‚Äîmilestones, phases grouped the way the cockpit expects. Think ‚Äúwhat streams exist over time.‚Äù |
| **`TASK-REGISTRY.xml`** | **Tasks** ‚Äî goals, keywords, verification commands, `@_status`. |
| **`STATE.xml`** | **Pointer** ‚Äî `current-phase`, `next-action`, optional agent registry. |
| **[documentation/requirements](/docs/documentation/requirements)** + **[documentation/state](/docs/documentation/state)** | **Monorepo narrative + cross-cutting queue** (canonical). Stub **`REQUIREMENTS.xml`** in `.planning/` points at these paths for XML-only tools. Repo-root **`REQUIREMENTS.md`** is a short pointer only. |
| **`.planning-reports/`** | **Local only** ‚Äî CLI usage + generated reports when **`REPOPLANNER_REPORTS_DIR`** / root **`pnpm`** scripts are used. |

**Phases folder** exists only after **`pnpm planning:init:full`** ‚Äî default minimal layout has **no** `.planning/phases/`.

## Cockpit product direction (planned)

The embedded **Repo Planner** UI at [`/docs/apps/repo-planner`](/docs/apps/repo-planner) should **not** depend on the planning CLI for normal use. Planned capabilities (see [task registry](/docs/repo-planner/task-registry) **`repo-planner-integration-01-12`**):

| Capability | Intent |
| --- | --- |
| **Multi-project** | Work with **several named projects** in one browser session (each with its own paths / logical workspace), without requiring a single fixed `REPOPLANNER_PROJECT_ROOT` for every action. |
| **Planning packs** | **Upload** or attach **planning packs** (exported `.md`, `.planning` zips, or similar) **alongside** the live repo snapshot ‚Äî view and edit **in memory** in the UI. |
| **localStorage persistence** | Persist **cockpit-only** state (open projects, pack metadata, UI layout, unsaved draft edits) in **`localStorage`** so refresh/tab close does not lose in-progress work before **export ‚Üí git**. |

**Shipped (MVP) on [`/docs/apps/repo-planner`](/docs/apps/repo-planner):** workspace sidebar (**This repository** vs uploaded file packs), **`localStorage`** for that workspace, **Export workspace JSON**, and embedded **`PlanningCockpit`** for the live repo (client dynamic import from `vendor/repo-planner`). **Not yet:** multiple server-backed project roots from the browser, or zip unpack in the UI (use multi-file pick for now).

Operators still rely on **committed `.planning/`** + **GET** read APIs for the live project; CLI remains optional for snapshots and checklists.

**Slimming `.planning`:** Target ‚Äúno more files here than you‚Äôd put in a **zip pack** you‚Äôd open in RepoPlanner.‚Äù Doc-template fragments (atoms/molecules/organisms) and long prose guidelines belong in **AGENTS.md**, **skills**, or **docs MDX**‚Äînot as extra top-level `.planning` artifacts. Trim in a dedicated task after confirming what the CLI requires at init.
