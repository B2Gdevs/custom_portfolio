# Repo Planner — Getting started

You do not have to use this monorepo’s layout to benefit from a **planning loop**: keep a visible **state**, an ordered **task registry**, and a **roadmap** (or phase list). Agents and humans read the same files; ids stay stable in chat and commits.

## Reference implementation

[MagicbornStudios/RepoPlanner](https://github.com/MagicbornStudios/RepoPlanner) provides:

- **XML-shaped** templates under `.planning/templates/` (bundled inside the submodule).
- **`loop-cli.mjs`** — snapshot, task updates, reports, `setup checklist`, `init` for greenfield bootstrap.
- Optional **cockpit UI** when embedded in a Next app (see upstream **INSTALL.md**). This portfolio keeps the **full editing cockpit local / operator-only**; production **docs** stay read-only.

## In this repo (from monorepo root)

```bash
git submodule update --init --recursive vendor/repo-planner
pnpm planning setup checklist    # verify .planning + XML when present
pnpm planning snapshot         # agent-friendly bundle / reports
pnpm planning:ui               # local markdown report viewer (127.0.0.1)
```

Pass **`--root <path>`** or set **`REPOPLANNER_PROJECT_ROOT`** if the CLI must resolve a different project root.

Greenfield bootstrap (creates XML + templates; use **`--no-agents-md`** here because **`AGENTS.md` already exists**):

```bash
pnpm planning:init
```

**Brownfield:** **`pnpm planning:init`** (minimal) creates **`.planning/`** for **XML** + **`planning-config.toml`** + **`.planning/AGENTS.md`**; may create repo-root **`REQUIREMENTS.md`** as a **stub** if missing. **Canonical** monorepo narrative: **[documentation/requirements](/docs/documentation/requirements)** + **[documentation/state](/docs/documentation/state)**. Execution lives in **`ROADMAP.xml`** / **`TASK-REGISTRY.xml`** / **`STATE.xml`** — not `IMPLEMENTATION_PLAN.md`. Stub **`REQUIREMENTS.xml`** points at those documentation paths; see [Integration](/docs/repo-planner/integration).
