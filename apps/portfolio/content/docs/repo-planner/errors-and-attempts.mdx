# Repo Planner -- Errors and Attempts

| When | Symptom | Cause | Mitigation |
| --- | --- | --- | --- |
| `2026-03-29` | **Client-side exception** on **`/apps/repo-planner`** (and modal) when a **pack workspace** was selected | **`localStorage`** workspace JSON could contain **`packs`** entries where **`files`** was not an array (hand edit, bad import, or future schema drift). Dashboard called **`activePack.files.map`** on bad data | **`parseWorkspaceState`** sanitizes packs, drops orphans, resets **`activeProjectId`**; **`Array.isArray`** before map -- **`lib/repo-planner-workspace-storage.ts`** |

## Embedded cockpit troubleshooting

| Symptom | What to do |
| --- | --- |

| **POST** apply / CLI returns **403** | Expected on public embed -- set **`PLANNING_COCKPIT_ALLOW_SERVER_WRITES=1`** only in trusted operator sandboxes ([decisions](/docs/repo-planner/decisions) **`REPOPLAN-EMBED-01`**) |
| No live data in production | Confirm **`.planning/`** is in the deploy artifact and the host allows short-lived **`spawn`** for read APIs ([planning-docs](/docs/repo-planner/planning-docs) **Host embed**) |

## Minimal init -- what went wrong once

An early **`--minimal`** pass still wrote **`REQUIREMENTS.md`** / **`IMPLEMENTATION_PLAN.md`** *inside* **`.planning/`**, breaking the bare tree. **`usage.jsonl`** also recreated **`.planning/reports/`**. **Fixes:** **`runPlanningInitMinimal`**, **`REPOPLANNER_REPORTS_DIR`**, **`scripts/run-planning-cli.mjs`** defaulting reports to **`.planning-reports/`** -- full table in [getting started](/docs/repo-planner/getting-started) and [decisions](/docs/repo-planner/decisions) **`REPOPLAN-REPORTS-01`**.

## Operator recovery

| Action | When |
| --- | --- |
| Clear **`repo-planner-workspace-v1`** in devtools **Application → Local Storage** | Cockpit misbehaves after a deploy (rare); forces default **This repository** workspace |
| Use **Export workspace JSON** before experiments | Restorable backup if you hand-edit storage |
