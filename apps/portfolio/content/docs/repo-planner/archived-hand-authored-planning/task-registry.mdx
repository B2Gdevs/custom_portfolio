## Phase `repo-planner-integration-03` (vendor cockpit body + data plane + shell upstream)

**Problem today:** [`/apps/repo-planner`](/apps/repo-planner) uses a **placeholder** live pane; workspace chrome is still **host-owned** (`PlanningCockpitDashboard` / `RepoPlannerWorkspaceShell`) while **`REPOPLAN-SHELL-01`** says **upload / pack switcher / export / init downloads** belong in the **`repo-planner`** package long term.

**Desired:**

- **Vendor `PlanningCockpit`** depth (metrics, %, charts, tasks, phases, state) composed in the live pane -- source: [`planning-cockpit.tsx`](https://github.com/MagicbornStudios/RepoPlanner/blob/main/components/organisms/planning-cockpit.tsx).
- **Host-mountable workspace shell** exported from **`repo-planner`** so the portfolio page is a thin **import + layout** (interim host shell until then).
- **Display** can use **parsed XML / on-disk reports** where possible so the UI is not defined by CLI spawn per poll for **read**; **writes** stay gated ([decisions](/docs/repo-planner/planning/decisions) **`REPOPLAN-EMBED-01`**).
- **Pack mode** feeds the **same bundle shape** as live repo (parse `TASK-REGISTRY.xml`, `STATE.xml`, `ROADMAP.xml`, report `.md` in pack).
- **Init template downloads** + **planning pack modal** copy per **`REPOPLAN-PACK-MODAL-01`**.

### Data contract (conceptual)

Vendor UI expects a **bundle** (see `Bundle` in `planning-cockpit.tsx`): `snapshot`, `openTasks`, `openQuestions`, `context.summary.phases`, `agentsWithTasks`, etc. Host implements something like **`CockpitDataSource`**: `getBundle()`, `getMetrics()`, `getLatestReport()`, polling policy.

| Source | `getBundle()` | `getMetrics()` | Report markdown |
| --- | --- | --- | --- |
| **Live monorepo** | `/api/planning-state` (today: CLI JSON) → later XML reader route | `/api/planning-metrics` | `/api/planning-reports/latest` |
| **Uploaded pack** | Client or **`POST /api/planning-pack/parse`** → same JSON shape | Synthetic series or honest empty | First `*.md` in pack or summary from XML |

**Resolved in phase `03`:** Pack mode now builds a cockpit bundle from XML/markdown planning files and exposes honest single-point metrics with no fake historical series.

### Phase `repo-planner-integration-05` notes

- Snapshot output should rank phases consistently across CLI and cockpit, so the scoring model, warning weights, stale thresholds, weighted effort, shrinking Global penalty, test-followup boost, timestamp tie-breakers, and open-question entropy penalty need explicit tests instead of hidden heuristics.

- **Planning inputs** never imply a blind scan of the whole repo: only config-declared roots + discovery rules (**`REPOPLAN-ROOTS-01`**).
- Snapshot scope is open work only: active and planned phases, plus current sprint context.
- The kickoff helper now has both a computed CLI surface and a committed example plan under `planning/plans/repo-planner-integration-05/PLAN.mdx`.

### Phase `repo-planner-integration-06` notes

- Forms should support visible phase open questions even when they do not block execution.
- Questions use a single text field.
- Question states are phase-only and limited to `open` and `answered`; answered items stay visible in a collapsed history area.
- Questions stay in created order.
- Each open question contributes equally to entropy. Two or more open questions should surface a `needs discussion` signal that lowers implementation recommendation and changes the suggested action toward discussion.
- Raw source stays hidden in RepoPlanner's normal product surface.
- Phase-question editing now ships on `-PLAN.xml` inspectors, and the pack-only migration flow is intentionally limited to stable string ids rather than short numeric/root ids.
- `workType` remains open for now; do not hard-code it until the host policy is settled.
