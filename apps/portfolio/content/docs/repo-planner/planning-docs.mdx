## Workflow stance

- **Roadmap, state, task registry, and decisions** are the living records.
- Any one-off phase plan should live under a section planning folder such as `planning/plans/<phase-id>/`.
- `IMPLEMENTATION_PLAN.md` should be treated as a legacy compatibility pointer only, not the default planning artifact for new work.
- RepoPlanner should align with the same structure instead of encouraging parallel "plan vs task registry" workflows.

## Host embed on this site

**Intent:** Browser edits stay **local / export → git**; the server does not persist casual cockpit mutations. Full detail: [Decisions](/docs/repo-planner/decisions) **`REPOPLAN-EMBED-01`**, **`REPOPLAN-SHELL-01`**.

| Surface | Default behavior |
| --- | --- |
| **GET** planning APIs | Read committed **`.planning/`** (today often CLI-backed JSON; XML read path planned -- [task registry](/docs/repo-planner/task-registry) **`03-03`**) |
| **`POST /api/planning-edits/apply`**, **`POST /api/planning-cli/run`** | **403** unless **`PLANNING_COCKPIT_ALLOW_SERVER_WRITES=1`** |
| **Planning chat / in-site LLM** | **Out of scope** -- **`POST /api/ai/planning-chat`** returns **501** unless you wire a backend |
| **Workspace chrome** (project switcher, pack upload, export, init downloads) | **Target:** owned by the **`repo-planner`** package and imported; portfolio only mounts and supplies routes/env (**`REPOPLAN-SHELL-01`**). **Today:** interim implementation still lives under **`apps/portfolio`** until upstream exports a host shell. |

### Production notes

**GET** handlers may **`spawn`** the CLI for **`simulate loop --json`** -- confirm the host allows short-lived processes and that **`.planning/`** is present in the deploy artifact if you want live data. With default **read-only** POST policy, HTTP does not write **`.planning/`**.
