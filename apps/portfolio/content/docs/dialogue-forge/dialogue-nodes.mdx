# Dialogue Nodes

Every beat in Dialogue Forge is a **node** in a `DialogueTree`. The **canonical** TypeScript definitions are in `packages/dialogue-forge/src/types` (see [Architecture](/docs/dialogue-forge/architecture)); the snippets here are for reading, not a second source of truth.

You normally create and connect nodes in **Graph** mode; the shapes below match what gets stored on disk.

## Node types

The shipped model uses **`npc`**, **`player`**, and **`conditional`** (`NODE_TYPE` in `packages/dialogue-forge`).

### NPC (`type: 'npc'`)

Character or narrator lines. Typically connects forward with `nextNodeId`, or into a player node for choices.

```typescript
{
  id: 'greeting',
  type: 'npc',
  speaker: 'Stranger',
  content: "Hello, traveler. What brings you here?",
  x: 0,
  y: 0,
  nextNodeId: 'first_choice'
}
```

**Common fields:** `id`, `content`, `speaker` or `characterId`, `nextNodeId`, `setFlags`, `x` / `y` for layout.

### Player (`type: 'player'`)

Player choice hub: **`choices`** array; each choice can branch with `nextNodeId`, set flags, and use **`conditions`**.

```typescript
{
  id: 'first_choice',
  type: 'player',
  content: '',
  x: 200,
  y: 0,
  choices: [
    {
      id: 'choice_friendly',
      text: "I come in peace.",
      nextNodeId: 'friendly_response',
      setFlags: ['was_friendly']
    },
    {
      id: 'choice_hostile',
      text: "None of your business.",
      nextNodeId: 'hostile_response',
      setFlags: ['was_hostile']
    }
  ]
}
```

### Conditional (`type: 'conditional'`)

Branches using **`conditionalBlocks`** (if / elseif / else style), for richer logic than a single player node.

Use the graph editor to author these; the exact block shape is defined in `packages/dialogue-forge/src/types`.

## Choices

Each choice can include:

| Property | Role |
| --- | --- |
| `id` | Stable id |
| `text` | Label the player sees |
| `nextNodeId` | Target node (optional if the choice ends the dialogue) |
| `setFlags` | Flags to set when taken |
| `conditions` | When this choice is visible |

## Conditions

`Condition` uses a **`flag`**, an **`operator`** (`is_set`, `is_not_set`, comparisons, etc.—see `CONDITION_OPERATOR` in the package), and optional **`value`** for comparisons.

Examples (conceptual):

```typescript
conditions: [{ flag: 'has_map', operator: 'is_set' }]
conditions: [{ flag: 'gold', operator: 'greater_equal', value: 100 }]
```

All conditions on a choice must pass (**AND**).

## Memory flags

Same ideas as before: set flags on NPC nodes or choices, gate choices with `conditions`, and keep names readable (`quest_dragon_accepted`).

## Flow patterns

**Linear:** NPC → NPC → … via `nextNodeId`.

**Branch:** NPC → player node with multiple `choices`, each with its own `nextNodeId`.

**Merge:** several branches set `nextNodeId` (or choice targets) to the same downstream NPC id.

**Loops:** a choice points back to an earlier id; avoid infinite loops with care.

## Endings

When nothing points forward and the playthrough reaches a leaf, the flow ends; Play mode surfaces that as end-of-dialogue behavior from the package.

## Tips

1. Use descriptive **`id`** values.
2. **Play** mode is the fastest sanity check for dead ends and unreachable nodes.
3. **Yarn** mode is ideal for proofreading line text and jump names before export.
