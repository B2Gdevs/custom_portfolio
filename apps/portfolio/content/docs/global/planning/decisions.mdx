# Decisions

| Id | Status | Decision | Impact |
| --- | --- | --- | --- |
| `global-site-dec-01` | `accepted` | **Global planning guide** (`global-planning.mdx`) is a **reference** doc, not part of the collapsed **planning** subtree. The Global planning section itself remains a normal section with requirements, roadmap, state, task-registry, decisions, and a compact playbook. | Sidebar keeps `global/planning/*.mdx` and `global-planning.mdx` distinct without hiding the Global section's planning records. |
| `global-site-dec-02` | `accepted` | The **Global** section is the human-first planning entrypoint for this repo; agents read it before picking a section or consulting `.planning/*.xml`. | Humans and agents share one visible workflow entrypoint on the docs site. |
| `global-site-dec-03` | `accepted` | New, ambiguous, or stale phases should use a lightweight kickoff record before broad implementation starts. | Agents must clarify scope, non-goals, risks, tests, and first tasks before doing large work. |
| `global-site-dec-04` | `accepted` | Executable behavior is not `done` without automated tests. Docs-only or content-only phases still need concrete verification commands, but they cannot waive test requirements for shipped behavior. | Definition of done now includes tests as a hard gate for implementation phases. |
| `global-auth-dec-01` | `accepted` | The site has no public signup in v1. One owner account and one tenant/org are seeded locally, anonymous visitors may still browse public surfaces, and gated features are unlocked by shared auth plus entitlements rather than section-specific passwords. | Books and Listen can add private or write-capable features without inventing separate access models. |
| `global-data-dec-01` | `accepted` | Shared app data and uploaded assets should move behind a provider contract that keeps local SQLite plus local files as the default development path while adding Supabase as the first hosted backend; the current auth/session layer stays separate for the first pass. | The repo no longer needs committed `.db` artifacts or Git LFS to share runtime state, and section planners can target one backend-neutral contract instead of forking persistence stories. |
| `global-data-dec-02` | `accepted` | The first hosted Supabase bootstrap uses one linked project ref, a private `reader-library` storage bucket, env-driven Payload Postgres/S3 settings in the repo, and a machine-local Codex MCP registration; only the database password and S3 access-key pair stay manual secrets. | Hosted setup becomes mostly mechanical instead of an undocumented dashboard ritual, while local SQLite/file mode remains available when the hosted secrets are absent. |
| `global-data-dec-03` | `accepted` | `@portfolio/repub-builder` should stop living as a normal package directory inside the portfolio monorepo and instead become its own standalone git-installable repo, vendored back into the monorepo as a submodule workspace dependency. | The package can evolve like a real external library, the monorepo keeps a vendored known-good checkout, and future consumers can install from GitHub without depending on this app repo layout. |
| `global-data-dec-04` | `accepted` | Published book binaries should move to storage as immutable versioned artifacts: each build can publish an EPUB plus a zipped planning pack under a version tag, Payload tracks the current version, and `public/books/manifest.json` keeps only lightweight metadata plus the current remote URLs. | Repo history stops being the normal home for shipped EPUB/zip binaries, checkpoints stay explicit, and SSR/SEO-facing routes can still render deterministic metadata without reading heavy files from git. |
| `global-data-dec-05` | `accepted` | Public SEO surfaces may be repo-first or Postgres-backed, but they must render from stable server-side slug and metadata loaders. Supabase storage is for heavy binaries, originals, downloadable bundles, and private uploads; it is not the canonical source for crawlable page copy. | The repo can move heavy files and mutable catalogs off git without giving up SSR or turning public content into client-only fetches. |
| `global-data-dec-06` | `accepted` | Public site media should use a lightweight manifest bridge during migration: upload the heavy files into `site-media-assets`, write a small manifest that maps legacy `/images/...` paths to storage-backed URLs, and resolve those paths inside the content loaders so authored MDX/frontmatter can keep their stable references while the binary files move out of the main runtime path. | The app can adopt storage-backed media without rewriting every existing project/blog document at once, and the same pattern can be extended to future public-media classes or retired later if content moves fully DB-backed. |
| `global-data-dec-07` | `accepted` | The next public catalog migration should separate metadata from heavyweight files: `/apps`, `/projects`, and `/resumes` may move their ordering, summaries, links, and download relations into Postgres collections (`site-app-records`, `project-records`, `resume-records`), while the actual downloadable or media-heavy artifacts move into storage collections such as `site-download-assets`. Long-form editorial bodies can stay repo-first until moving them would truly reduce maintenance. | This keeps SSR and SEO strong, shrinks the repo's file and binary burden, and avoids forcing every public content surface into the database before the metadata and storage split is paying for itself. |
| `global-data-dec-08` | `accepted` | Public planning-pack exports and resume HTML artifacts should publish through the shared `site-download-assets` collection, and site consumers should prefer those published asset URLs while keeping repo-local files as a fallback until metadata migrations are complete. Because the collection is script-published only, it may allow restricted file types such as HTML and markdown without turning into a public upload surface. | One generic download pipeline now covers planning-pack files, resume artifacts, and future public bundles without bespoke buckets or route-specific storage rules. |
| `global-auth-dec-02` | `accepted` | v1 site auth remains owner-only even as the shell is refreshed: public visitors browse almost everything anonymously, but login UI should only unlock private/admin surfaces for the single seeded owner account configured by `PORTFOLIO_OWNER_EMAIL` and related env vars until a broader role model is intentionally added. | The sidebar/login revamp can reuse the existing auth/session API and shadcn-compatible UI patterns without opening public signup or implying multi-user product behavior the backend does not support yet. |
