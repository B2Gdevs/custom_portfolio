# mb-cli-framework — shared CLI shell

## Problem

We currently maintain **two** operator CLIs (**`magicborn`** in this repo, **`grimetime`** in Grime Time) that should share:

- **Predictable namespacing** (portfolio vs vendor-forward vs top-level vendor topics).
- **Tab-first navigation** as the primary DX (completion lists, discoverable subcommands).
- **Visible scope** when “using” a vendor workspace (like `git` branch context), with a **clear exit**.
- **Optional color** in completions for categories (e.g. “has controllable CLI” vs read-only), respecting **`NO_COLOR`** / **`FORCE_COLOR`**.
- A path toward **generated** commands for **OpenAI**, **local RAG**, and **Payload CMS** surfaces, discovered at **install** or **runtime** from each app’s config.

We do **not** want two diverging frameworks.

## Direction

| Decision | Choice |
| --- | --- |
| **Source of truth** | Git submodule **`vendor/mb-cli-framework`** → [github.com/MagicbornStudios/mb-cli-framework](https://github.com/MagicbornStudios/mb-cli-framework) |
| **Consumers** | **Portfolio:** `@magicborn/cli` + `apps/portfolio/scripts/magicborn/*`. **Grime Time:** `bin/grimetime.mjs` + `src/cli/*` (same patterns, different binary name). |
| **Contract** | TypeScript package exporting: completion helpers (bash/zsh/fish snippets or generators), **scope** helpers (`use` / `clear` env contract), **color** tokens, optional **Payload/OpenAI/RAG** discovery interfaces (implemented incrementally). |
| **Vendor UX** | `magicborn vendor` second-token completion lists **registered vendor ids** with a **suffix** (e.g. `(cli)` when `.magicborn/cli.toml` declares a controllable bin). **Colors:** opt-in via env; document terminal limitations (bash `compgen` is plain-text first; richer UX in zsh/fish). |
| **Scope / become** | **`magicborn vendor use <id>`** (or `enter`) sets **`MAGICBORN_VENDOR_ID`** (and optionally **`MAGICBORN_VENDOR_ROOT`**) for the current shell session; **`magicborn vendor clear`** unsets. Optional **`MAGICBORN_PS1=1`** documents a **prompt snippet** (git-style) — not mandatory for CI. |
| **Autogen commands** | **Phase 2:** introspect Payload `payload.config` / collection list + env for OpenAI + RAG config; generate or cache command stubs under **`.magicborn/commands/`** or register at runtime. Assumes “cheap local LLM” RAG and OpenAI are **defaults** but overridable per project. |

## Non-goals (this phase)

- Replacing **Citty** or **Commander** entirely — adapters can wrap both.
- Full Payload CRUD codegen before the **discovery spike** is validated.
- Perfect color in **all** terminals — spec **degrades gracefully** to suffix letters: `(c)` = controllable CLI, `(·)` = manifest only.

## External operator CLIs (Claude Code, Codex)

**Portfolio `magicborn` does not launch `claude` or `codex`.** Install those CLIs separately, use **`cdmb`** (from **`shell-init`**) or **`cd "$MAGICBORN_REPO"`**, then run **`claude`** / **`codex`** in the shell. Rationale: **Node-spawned** children break **TTY / `--print`** detection on some Windows setups; supporting that inside `magicborn` is **out of scope** for now (**[global-tooling-dec-06](/docs/global/planning/decisions)**).

**Framework consumers** (e.g. Grime Time) may still use **`@magicborn/mb-cli-framework`** helpers **`runExternalCliViaCmdCdExec`**, **`runExternalCliViaMsysBashCdExec`**, **`shouldUseMsysBashExecWrapper`** if they choose to implement their own forwarders. Historical attempts: **[errors-and-attempts](/docs/global/planning/errors-and-attempts)**, **[mb-cli-windows-operator-forward KICKOFF](/docs/global/planning/plans/mb-cli-windows-operator-forward/KICKOFF)** (archived context).

## Payload dev database sandbox (planned)

**Goal:** one **`magicborn` / framework** command family to **restore production backups** into a **local-isolated** database, with an env-driven **sandbox mode** that blocks **outbound integrations** (webhooks, email, third-party sync) while keeping Payload schema and admin usable for debugging.

**Standard collections / metadata** (optional framework slice): backup job records, environment labels, or restore checkpoints — only where they help **multiple apps**; avoid duplicating Payload migrations.

**Kickoff:** [PAYLOAD-MCP-CHAT-KICKOFF](/docs/global/planning/plans/mb-cli-framework/PAYLOAD-MCP-CHAT-KICKOFF) · Tasks: **`global-tooling-09`** in [task registry](/docs/global/planning/task-registry).

## Extension convention (planned)

CLI features that are **not** pure Payload CRUD (e.g. **books**: EPUB, scene extract, file generators) register via a **documented convention** (e.g. **`.magicborn/cli.toml`** or workspace **`magicbornTopics`** export) so the core shell stays thin and **section-specific** logic stays composable.

## Shell init safety (Windows / Git Bash)

- **`~/.bashrc` must not run Node** to load completions (no `eval "$(magicborn completion bash)"` during rc). That subprocess loads the full CLI while stdout is a pipe and has **hung Git Bash before login prompt** in the wild.
- **`magicborn shell-init bash --apply`** should **`source` the static file** `packages/magicborn-cli/completions/magicborn.bash` from `MAGICBORN_REPO` after exporting that variable. Tab completion still shells out to `magicborn __complete …` only when you press Tab.
- **`magicborn.bash`** must avoid `git rev-parse` when `MAGICBORN_REPO` is set but workspace completion files are missing (same hang class from `$HOME` or odd `PWD`).

## Future — native binaries + installer (magicborn, grimetime, framework consumers)

Goal: **ship a single executable + small installer** (or portable zip) so operators are not dependent on **shell hooks**, **global npm link**, or **Node in PATH** for day-to-day use. Shell integration becomes optional.

| Track | Notes |
| --- | --- |
| **Build** | **pkg**, **nexe**, or **Node SEA** (single-file executable) for `magicborn` / `grimetime`; bundle `dist/` + static completions. |
| **Installer** | Windows: user-scope PATH + Start Menu; macOS: `.pkg` or Homebrew cask; keep **idempotent** upgrade. |
| **Framework** | `mb-cli-framework` documents a **consumer checklist**: plain-mode defaults for CI, **no TUI on piped stdout**, completion assets as **static files** where possible. |
| **Grime Time** | Mirror the same distribution story for `grimetime` (vendor repo); align version + signing policy with portfolio. |

Planning slices: add tasks under **`global-tooling`** (or a new **`global-tooling-0x`**) when this moves from ADR to implementation; until then this section is the **intent**.
