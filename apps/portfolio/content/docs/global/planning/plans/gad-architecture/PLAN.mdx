# get-anything-done — architecture, boundary spec, and migration

## Summary

**get-anything-done (GAD)** is a direct replacement for GSD when installed. It is powered by GSD as a peer dependency — GSD is the planning engine, GAD is the platform layer that makes it scale across monorepos, teams, and products.

**repo-planner (RP)** is the official GUI for GAD.
**mb-cli-framework** powers the CLIs for GAD, portfolio, and grime-time.

> RP was the previous implementation of the GAD workflow. Skills (`rp-*`) encoded the methodology manually. GAD is the proper home for that work — evaluated and refined continuously via the eval framework.

### Why GAD instead of GSD

| GSD | GAD |
|---|---|
| One project at a time | Monorepo: N projects, one view |
| Internal `gsd-tools.cjs` only | `gad` public CLI — humans + tooling + web apps |
| No extension system | `.gad` domain packs |
| No eval/drift framework | TRACE + SCORE + benchmark runs |
| No docs output | Docs sink → MDX for display |

GAD installs GSD first, then replaces all `gsd:` commands with `gad:`. **One prefix, no confusion.**

### Scaling model

```
Simple (1 repo)     → install GAD, run gad:new-project, works like GSD
Monorepo            → add [[planning.roots]] entries, gad tasks across all roots
Ecosystem           → .gad domain packs, eval framework, docs sink to portfolio
```

## Boundary spec

### What lives in GAD (methodology, no GUI, portable)

- All planning commands (`gad discuss-phase`, `gad plan-phase`, `gad execute-phase`, etc.)
- All subagents (`gsd-planner`, `gsd-executor`, `gsd-verifier`, etc.)
- State / roadmap / requirements templates (Markdown — `STATE.md`, `ROADMAP.md`, `REQUIREMENTS.md`)
- Config system (`config.json` with `planning.sub_repos`, `planning.ignore`, `planning.docs_sink`)
- Sprint model (version-cycle-based, configurable up to N sprints per milestone — default 5, overridable in `planning-config.toml`)
- Monorepo / vendor extensions (vendor directory policy, multi-`.planning/` awareness)
- Migration system (parallel to GSD `update` command — `gad update` + migration scripts)
- `.gad` pack spec and loader
- GAD CLI (`gad` binary — see `gad-cli` plan)
- Eval framework and drift metrics (see `gad-eval` plan)

### What lives in repo-planner (GUI only)

- Web UI (Payload CMS + Next.js)
- Planning file discovery across vendor submodules
- `MD → MDX` transformation and documentation routing (docs sink compiler)
- Visual phase / roadmap / task dashboard
- `rp-manuscript` as a RP-specific `.gad` pack (fiction/creative writing domain)
- Any skill that wraps a web API or UI action specific to the portfolio app

### What gets removed from repo-planner (GAD owns these)

All the following RP skills are redundant — GAD has direct equivalents:

| Remove from RP | GAD equivalent |
| --- | --- |
| `rp-add-todo` | `gad add-todo` |
| `rp-check-todos` | `gad check-todos` |
| `rp-debug` | `gad debug` |
| `rp-execute-phase` | `gad execute-phase` |
| `rp-map-codebase` | `gad map-codebase` |

## Schema migration: XML → Markdown

RP currently uses `STATE.xml`, `ROADMAP.xml`, `TASK-REGISTRY.xml`, `REQUIREMENTS.xml`.
GAD uses `STATE.md`, `ROADMAP.md`, `REQUIREMENTS.md` (Markdown).

**Migration path:**

1. Write a `gad migrate-schema` command that reads RP XML files and outputs GAD MD equivalents
2. The portfolio's existing `.planning/*.xml` files stay as-is until the migration command runs
3. After migration, the XML files are archived (not deleted) for reference
4. The docs sink (MDX compiler) targets the MD format going forward

**Schema mapping:**

| RP XML | GAD Markdown |
| --- | --- |
| `STATE.xml` → `<current-phase>` | `STATE.md` → `## Current Position / Phase: X of Y` |
| `ROADMAP.xml` → `<phase id>` | `ROADMAP.md` → `- [ ] **Phase X: Name**` |
| `TASK-REGISTRY.xml` → task rows | `STATE.md` → `## Performance Metrics` + phase tables |
| `REQUIREMENTS.xml` → stub pointers | `REQUIREMENTS.md` → full requirements prose |
| `DECISIONS.xml` | stays in MDX public docs (no XML equivalent in GAD) |

## Docs sink architecture

```
Layer 1 — Working .planning/ dirs (source of truth, per-project)
  vendor/get-anything-done/.planning/
  vendor/repo-planner/.planning/
  .planning/                              ← monorepo cross-cutting
  vendor/grime-time-site/.planning/
  ...

Layer 2 — The sink (compiled, public-facing, read-only)
  apps/portfolio/content/docs/planning/
  ├── get-anything-done/
  │   ├── STATE.mdx
  │   ├── ROADMAP.mdx
  │   └── phases/
  ├── repo-planner/
  └── grime-time-site/

Layer 3 — Config (in root config.json or planning-config.toml)
  planning.docs_sink = "apps/portfolio/content/docs/planning"
  planning.sink_workflow = "manual"   ← explicit: run `gad docs compile`
```

`gad docs compile [--sink <path>]` reads all discovered `.planning/` dirs → transforms to MDX → writes to sink. Portfolio docs site renders from sink. **Sink is never the source.**

### Sink workflow

`sink_workflow` options: `manual` (default, explicit compile) | `compile-on-phase-complete` | `compile-on-state-change`.

## .gad pack format

A `.gad` file is a planning pack — a versioned, distributable bundle of GAD extensions for a specific domain (e.g. fiction writing, API development, web app).

### Directory structure (unpacked)

```
my-pack.gad/
├── gad.json              ← pack manifest
├── commands/             ← slash commands extending GAD
├── agents/               ← subagent prompts
├── workflows/            ← workflow documents
├── templates/            ← file templates (STATE, PLAN, KICKOFF, etc.)
└── references/           ← planning reference docs
```

### `gad.json` manifest schema

```json
{
  "name": "manuscript",
  "version": "1.0.0",
  "extends": "get-anything-done",
  "min_gad_version": "1.31.0",
  "description": "Fiction/creative writing planning pack",
  "domain": "creative-writing",
  "entry_commands": ["gad:manuscript-new", "gad:manuscript-chapter"],
  "token_estimate": {
    "install_tokens": 4200,
    "per_phase_tokens": 1800
  },
  "state_extensions": {
    "canon_locks": [],
    "forward_questions": []
  },
  "install": {
    "commands": true,
    "agents": true,
    "templates": true,
    "workflows": true
  }
}
```

### Token meta

Each pack declares `token_estimate`:
- `install_tokens`: expected tokens consumed when the pack is first loaded into context
- `per_phase_tokens`: expected tokens per average phase execution with this pack

This data feeds the eval framework's efficiency scoring.

### Distribution

- `.gad` files are zip archives of the above directory structure
- Installed via `gad skills install <repo> --skill <pack-name>` (uses `npx skills add` pattern)
- Or from a local path: `gad skills install ./my-pack.gad`
- Registry of first-party packs lives at `MagicbornStudios/get-anything-done/packs/`

## planning-config.toml — canonical config format

The portfolio already has the right structure. GAD reads and extends this file:

```toml
[planning]
sprintSize = 5                        # configurable, default max 5 sprints per milestone
currentProfile = "human"
conventionsPaths = [
  "AGENTS.md",
  ".planning/AGENTS.md",
  "apps/portfolio/content/docs/documentation/requirements.mdx",
]
docs_sink = "apps/portfolio/content/docs/planning"   # GAD addition
ignore = ["**/node_modules/**", "**/dist/**"]         # GAD addition

[[planning.roots]]
id = "global"
path = "."
planningDir = ".planning"
discover = false

[[planning.roots]]
id = "grime-time"
path = "vendor/grime-time-site"
planningDir = ".planning"
discover = true

[profiles.human]
description = "Default human view."

[profiles.agent]
description = "Agent perspective; defaultJson = true."
defaultJson = true
```

`[[planning.roots]]` IS the multi-project registry. `discover = true` means auto-crawl. `gad workspace sync` populates this from crawl. GAD reads all roots for `gad tasks`, `gad state`, `gad docs compile`.

## Command set decisions (2026-04-03)

### Keep (GAD ships these)

| Command | Reason |
|---|---|
| All GSD core commands | Direct replacement — same workflow, `gad:` prefix |
| `autonomous` | **Default GAD workflow.** discuss→plan→execute per phase, handles full milestone loop. `--only N` for single phase, `--interactive` keeps discuss inline. |
| `validate-phase` | Retroactive Nyquist validation — fills test coverage gaps |
| `next`, `do`, `forensics`, `stats`, `session-report` | Automation + observability |
| `thread`, `note`, `plant-seed` | Context continuity across sessions |
| `manager` | Dashboard for milestone oversight |
| `discuss-phase`, `list-phase-assumptions` | Pre-planning quality |

### Remove from GAD (use CLI instead)

| Command | Reason |
|---|---|
| `workstreams` | Use `gad` CLI for parallel project views |
| `list-workspaces`, `new-workspace`, `remove-workspace` | Use `gad workspace *` CLI commands |
| `ui-phase`, `ui-review`, `secure-phase` | Domain-specific, not GAD core |
| `pr-branch`, `ship` | Use git/gh directly |
| `profile-user` | Deferred |

### Autonomous as default

`gad:autonomous` is the standard way to run a milestone. It is not a "special mode" — it is the expected agent invocation. When a coding agent is told to work, autonomous handles the discuss→plan→execute loop. `--interactive` is for human-in-the-loop sessions.

## Nyquist auditor

`gsd-nyquist-auditor` fills test coverage gaps after a phase completes. Named after Nyquist sampling theorem — every requirement needs at least one observable test to "sample" it faithfully.

**What it does:** For each requirement with no test: classifies test type (unit/integration/smoke), generates minimal behavioral test, runs it, debugs up to 3 iterations. Implementation files READ-ONLY — bugs escalated, never fixed.

**Why it matters for GAD:** Pairs directly with the eval framework. Document drift detection (leftover text in planning docs) + Nyquist coverage gaps are the two signals that tell us where the methodology has holes.

## Parallelism — open design question

GAD uses wave-based parallel execution in `execute-phase`. Concerns raised:

**Benefits of parallelism:**
- Faster wall-clock execution
- Each agent gets fresh full context window (200k tokens)
- Failures isolated per task

**Costs of parallelism:**
- Each agent re-reads all context files (multiplied token cost)
- Shared planning file conflicts (STATE.md, ROADMAP.md written by multiple agents)
- Skills activated even if not needed for the specific task
- Context loss: agents can't see each other's work mid-wave

**The key principle:** parallelize only **independent tasks with no shared write targets**. Sequential for anything touching STATE.md, ROADMAP.md, or files another task writes.

**For the eval framework:** measure this explicitly:
- `parallel_invocations_per_phase` — how many agents spawned
- `token_cost_per_agent` — context re-read overhead
- `conflict_rate` — how often agents write the same file
- `skill_activation_waste` — skills loaded but never invoked

High conflict rate or skill_activation_waste → the wave grouping needs refinement.

**Preferred model for GAD:** sequential by default, parallel only when the plan explicitly marks tasks as independent. Coding agents should prefer sequential unless the parallelism benefit is proven by eval data.
