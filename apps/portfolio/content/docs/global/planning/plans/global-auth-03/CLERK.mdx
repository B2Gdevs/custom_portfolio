# Why this note exists

Grime Time has now started the real Clerk migration, but the portfolio cannot simply copy that auth shape.

The portfolio has a different auth problem:

- tenants/orgs already exist in the planning model
- invites are part of the future shared auth foundation
- roles and entitlements need to span multiple product surfaces
- the site is still mostly public and owner-operated today

That means Clerk may be the right hosted identity layer, but the portfolio needs a planning-first pass before implementation.

## Working direction

- **Clerk** is the leading candidate for hosted identity, social login, and account UI.
- **Payload users + tenants + entitlements** remain the authorization core.
- invite acceptance should still be tenant-scoped and default-deny.
- section/product surfaces should continue asking the shared auth layer for permissions instead of talking directly to Clerk concepts.

## What the portfolio must answer first

1. How Clerk identities map onto existing tenant membership and entitlements.
2. Whether Clerk organizations are useful here, or whether tenant/org state should stay entirely first-party in Payload.
3. How invite acceptance should bind a Clerk identity to an existing Payload user record.
4. How owner-only today transitions into broader org access without weakening the current restricted-operator posture.
5. How future billing/customer ids, if they land in the portfolio stack, relate to the same external-id model used in Grime Time.

## Non-goal

Do not start swapping the portfolio auth/session code to Clerk until those multi-tenant answers are explicit.

The goal of `global-auth-03-04` is to define the migration contract, not to copy Grime Time's customer-portal auth surface into a different product.

## Answers to multi-tenant questions (2026-04-02)

### 1. How Clerk identities map onto existing tenant membership and entitlements

**Answer:** Clerk handles **authentication only**. When a Clerk user authenticates:

1. Clerk webhook (`user.created`, `user.updated`) fires
2. Webhook handler looks up or creates a Payload `users` record by `clerkId` (new field)
3. If user doesn't exist in Payload, create with:
   - `clerkId`: Clerk user ID
   - `email`: from Clerk
   - `displayName`: from Clerk profile
   - `avatarUrl`: from Clerk profile
   - `role`: `member` (default)
   - `tenant`: lookup from invite or default tenant
   - `entitlements`: empty (default-deny per `global-auth-dec-05`)
4. If user exists, update profile fields but preserve Payload-managed fields (`role`, `tenant`, `entitlements`)

**Entitlements stay in Payload.** Clerk never knows about `reader:sync`, `admin:access`, etc.

### 2. Whether Clerk organizations are useful here

**Answer:** **Use Clerk organizations sparingly.** The portfolio has:
- `tenants` collection in Payload (already exists)
- Multi-tenant membership model planned

Clerk orgs can:
- Provide org switcher UI
- Enable org-scoped sessions
- Simplify "which org am I in" for the user

But **Payload tenants remain source of truth**:
- Clerk org ID maps to Payload tenant ID via `clerkOrgId` field on tenants
- Membership changes in Clerk trigger webhooks that sync to Payload
- Authorization checks always hit Payload, never Clerk metadata

**Recommendation:** Enable Clerk organizations but treat them as a UI/session convenience, not the authority.

### 3. How invite acceptance binds Clerk identity to existing Payload user

**Answer:** Two flows:

**Flow A: Clerk-first signup (via invite link)**
1. Operator creates invite with `pnpm auth:invite` (generates Payload invite token + expected email)
2. Invite link goes to `/accept-invite?token=...`
3. Page checks token validity, shows tenant/role context
4. User clicks "Sign up with Clerk" -> redirected to Clerk signup
5. After Clerk signup, Clerk webhook fires with `user.created`
6. Webhook handler checks for pending invite by email
7. If invite exists and valid: bind `clerkId` to the pre-created Payload user, mark invite consumed
8. If no invite: create minimal Payload user with `member` role, no tenant (or default tenant)

**Flow B: Existing Payload user adds Clerk identity**
1. User with Payload-only auth logs in
2. User clicks "Link Clerk account" in settings
3. OAuth flow to Clerk
4. On success, update Payload user with `clerkId`

### 4. How owner-only transitions to broader org access

**Answer:** Phased rollout:
1. **Phase 1 (current):** Owner-only, Payload auth
2. **Phase 2 (this task):** Add Clerk as optional identity layer, owner can use Clerk login
3. **Phase 3:** Invites work with Clerk signup for `admin`/`member` roles
4. **Phase 4:** Full org management via Clerk org UI + Payload sync

**Session bridge:** Both auth methods produce the same session shape:
- Payload-only: existing `portfolio-token` cookie
- Clerk: Clerk session -> lookup Payload user by `clerkId` -> same entitlement checks

### 5. How billing/customer IDs relate to external-id model

**Answer:** Add `externalIds` field to users collection:
```typescript
{
  name: 'externalIds',
  type: 'group',
  fields: [
    { name: 'clerkId', type: 'text', unique: true, index: true },
    { name: 'stripeCustomerId', type: 'text', unique: true, index: true },
    // Future: other provider IDs
  ],
}
```

This keeps provider-specific IDs in one place and allows the same user to have Clerk identity + Stripe billing without coupling those systems.
