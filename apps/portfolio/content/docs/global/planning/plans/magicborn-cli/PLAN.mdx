# Magicborn CLI — is it a good idea?

**Yes**, for *this* repo: you already run **many** `pnpm` scripts (auth seed, RAG ingest, book build, media publish, provider bootstrap). A **single entry** with **consistent flags**, **`--help`**, **config discovery**, and **tabular output** reduces cognitive load and avoids duplicating Payload/env bootstrapping in one-off scripts.

## Recommended shape

| Principle | Why |
| --- | --- |
| **One binary, subcommands** | `magicborn users list`, `magicborn media prompts`, `magicborn books status` — matches mental model of `kubectl`, `flyctl`, `pnpm`. |
| **Thin wrappers first** | Delegate to existing `tsx` scripts and `lib/` until a command needs bespoke behavior. |
| **Shared runtime** | Reuse `loadScriptEnv`, `getPayloadClient`, logging — same as `invite-user` / `rag-ingest-runner`. |
| **JSON + table modes** | `--json` for automation; default human tables for terminal UX. |

## Candidate command groups (backlog)

| Group | Examples |
| --- | --- |
| **auth** | `users list`, `users get --email=`, `invite` (wrap `auth:invite`) |
| **books** | `books list`, `books missing-covers`, `books build`, `books publish` |
| **media** | `media prompt show`, `media generate --scene=…`, `media gaps` (records without `imagePrompt` metadata — needs schema) |
| **data** | `rag ingest`, `rag status`, `provider export` |
| **planning** | `planning packs list`, `planning embed` |

## Architecture notes

- Implement with **`commander`** or **`yargs`** + **`tsx`** shebang, or a small **`packages/magicborn-cli`** workspace package consumed as `pnpm mb`.
- **Max UX**: top-level `magicborn --help`, **per-command help**, **exit codes**, **NO_COLOR** / **FORCE_COLOR**, and **stable stderr** for errors.

### CLI visual language (terminal)

**Implemented in** `apps/portfolio/lib/magicborn/magicborn-cli-ui.ts` (when **`MAGICBORN_CLI=1`** and not **`--json`**).

| Role | Meaning | Default (color on) |
| --- | --- | --- |
| **Domain ribbon** | Same **families** as `magicborn --help`: asset/repo vs Payload catalog | **`book`**, **`planning-pack`**, **`listen`** → **yellow**; **`app`**, **`project`** → **cyan** |
| **Verb** | Primary action (`generate`, `Calling`, first word of a step) | Bold cyan |
| **Argument / value** | Models, paths, sizes | Yellow |
| **Flag names** | `--print-prompt`, labels like `prompt` / `manifest` in summaries | Magenta |
| **Meta / paths** | Repo path, hints, secondary copy | Dim |
| **Section rules** | Phase boundaries (`RAG · prompt context`, `OpenAI · image generation`) | Magenta `──` rules |
| **Info** | Neutral operator notes | Blue `ℹ` + dim on **stdout** |
| **Warnings / dry-run / prompt-only** | Advisory footer copy | Yellow + dim on **`stderr`** (`console.warn`) so they stay visible next to stdout transcripts |
| **Errors** | Failures | Red on **`stderr`** (`console.error`) |
| **Success** | Done block | Green `✓`; spend line yellow on stdout |
| **Long-running** | OpenAI image request | Braille spinner on TTY until the call returns |

**Planning packs** sit in the **asset & repo** family with **books** and **listen**: not Payload-only CRUD; they use **repo / embed assets** alongside CMS. **Yellow** ribbons and the **first help section** describe that.

### `magicborn --help` (grouped colors)

**Implemented in** `packages/magicborn-cli/src/help-format.ts` via Commander **`configureHelp`** (root command only; subcommands keep default help).

| Section title (bold) | Command name color | Commands |
| --- | --- | --- |
| Asset & repo (local + CMS, not Payload-only) | Yellow | `book`, `planning-pack`, `listen` |
| Payload CMS catalog | Cyan | `app`, `project`, `payload` |
| Vendor | Green | `vendor`, `users`, `org`, `tenant`, `blog` |
| OpenAI | Blue | `openai` |
| Model & style | Magenta | `model`, `style` |
| Shell, completion, updates | Gray | `completion`, `shell-init`, `update` |
| (body text) | Dim **magenta** | Descriptions in the right-hand column |

**Workspace:** `pnpm <args>` is listed separately (dim gray) — not a Commander subcommand.

**Open questions** (optional later): dedicated **stderr** stream for **info** lines; richer **subcommand** help (`magicborn book --help`) with the same palette.

### Tab completion layout (bash / zsh / fish)

- **Bash** uses GNU **readline**: matches are drawn in a **grid**; column order is controlled by **`print-completions-horizontally`**. **`magicborn shell-init bash`** sets **`off`** so read order is **column-major** (down the first column, then the next). True **one item per line** is **not** a bash default; use **zsh** `menu select` or **fish**’s pager for that feel.
- **`magicborn completion zsh`** / **`fish`** snippets document optional **`zstyle`** / pager color lines.

## Planning tasks

Tracked as **`global-tooling-01`** in [global task registry](/docs/global/planning/task-registry).

### Landed (v0)

- **Root:** `pnpm magicborn` / `pnpm mb` → `packages/magicborn-cli` (Commander), **`prepare`** builds `dist/` for the bin.
- **`generate`:** nested commands `book` \| `app` \| `planning-pack` \| `listen` → `apps/portfolio/scripts/magicborn/run.ts`. Shared **`lib/magicborn/openai-image-generate`** with **`POST /api/media/generate`**. **`.env`** via `load-magicborn-env` (no `@next/env` in tsx). Output: **`.magicborn/runs/<id>/`** (`image.png`, `prompt.txt`, `manifest.json`). Flags: **`--dry-run`**, **`--print-prompt`**, **`--raw`**, **`--size`**, **`--slot`**, **`--json`**. Terminal UX mirrors **`rag:ingest`** when **`MAGICBORN_CLI=1`** (set by the package forwarder).
- **`vendor add`:** GitHub URL → dry-run instructions or **`--apply`**: `git submodule add` + **`pnpm-workspace.yaml`** entry (see **`vendor/repub-builder`**); npm/GitHub Packages notes point at root **`package:*`** scripts — no secrets in repo.

### Media asset taxonomy (operator + site)

Every generated file maps to a **stable ref** for CLI, local working files, and Payload/storage: **`kind` + record id (+ optional sub-id / variant)**. Kinds align with what the app already surfaces or will surface soon.

| Kind | Typical use | Text in image? | Notes |
| --- | --- | --- | --- |
| **`site`** | Logos, favicon-related art, shared chrome | **Logo:** usually no embedded tagline unless intentional; wordmarks are a separate deliberate choice | **Active logo** (and similar) is an **explicit pointer** set by the operator — not “latest upload wins” (**`global-tooling-dec-03`**). |
| **`book`** | Covers, scene stills, character sheets | **Covers:** yes — title + subtitle (and series if needed) per brief | Scene/character images often **no** burned-in text; captions live in CMS/copy. |
| **`app`** | Catalog cards, hero imagery for `/apps` | Often **yes** — app name + short subtitle for card-style art | Matches “book cover with text” energy for discoverability. |
| **`project`** | Project covers / thumbnails | Often **yes** — project title + optional tagline | Same pattern as apps where the grid is image-forward. |
| **`song`** / **`listen`** | Listen row art, BandLab-linked visuals | Case-by-case; default **no** text unless promoting a track title on the art | Keep flexible per row. |
| **`planning-pack`** | Pack thumbnails, modal art | Usually **no** or minimal; title in UI | Align with planning embed surfaces. |
| **`blog`** / **`article`** | Post heroes, OG-style images | Optional title on art for share cards; editorial choice | Not all posts need text on image. |
| **`ui`** | Backgrounds, textures, marketing shells | Typically **no** text | Style layer + composition only. |

**Prompt architecture (two layers):** (1) **Magicborn media design style** — shared art direction (repo `lib/magicborn-prompts` first; optional Payload override later). (2) **Asset prompt** — the specific scene, product, or layout brief. Image models that must **avoid** text still use the global negatives from the style block; when text **is** wanted, the asset prompt states typography intent explicitly.

### Next wave — Ink TUI, chat, vendor v2 (`global-tooling-03`)

The current **picocolors + string `--help`** path stays for **pipes, CI, and `NO_COLOR`**. The **default interactive** experience is planned to move to **Ink** (layouts, borders, list/detail, forms) with optional **[assistant-ui for Ink](https://www.assistant-ui.com/ink)** for guided flows (e.g. **`payload <alias> generate`** when fields are missing). **Vendor** help will list **discovered vendor ids** under **`vendor`** only (remove misleading top-level **`users`/`org`/…**). **Yellow** in the next design pass should track **local/repo actions** (packaging, publishers) as well as creative asset flows.
