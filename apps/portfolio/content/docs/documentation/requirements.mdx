## Core Requirements

| Id | Requirement | Why |
| --- | --- | --- |
| `DOC-REQ-01` | each active section should own its planning loop inside the docs tree | planning state should live next to the public implementation context |
| `DOC-REQ-02` | the docs area should expose a roadmap with section-level status and next steps | readers need a short overview before drilling into section records |
| `DOC-REQ-03` | planning docs and reference docs should remain clearly separated in section navigation | day-to-day planning should stay compact without burying reference material |
| `DOC-REQ-04` | docs copy should be compact, field-based, and easy to render into structured UI later | these pages are living records, not prose essays |
| `DOC-REQ-05` | section planning MDX is the first stop for **section** work; **root `.planning/*.xml`** + **root `AGENTS.md`** for **monorepo** gates -- same workflow, two layers | matches root [`AGENTS.md`](../../../../../AGENTS.md) |
| `DOC-REQ-06` | planning MDX may include `repoPath` and `taskPhase` frontmatter for local search; values must stay repo-accurate | agents and humans find files without guessing route-to-path mapping |
| `DOC-REQ-07` | planning pack modal: demo `.md` plus build-time exports of section planning MDX only; `.planning` not public; open from nav/hero via Zustand modal registry (`documentation-site-03-04`) | planning packs policy also summarized in [global planning](/docs/global/global-planning) |
| `DOC-REQ-08` | the docs IA should expose `magicborn` as a first-class section for public worldbuilding material written in short story form | world/lore writing needs a stable public home and a clear ingestion source for site chat |
| `DOC-REQ-09` | active public content areas like `blog` and `projects` should have section-local planning docs when their UX, IA, or schema is being actively redesigned | content-structure decisions should not get buried in unrelated documentation pages |
| `DOC-REQ-10` | operator-facing planning docs should prefer ASCII punctuation, while public-facing prose may keep richer Unicode typography | shell, CLI, XML, and generated planning artifacts should stay readable across terminals without flattening the site's published copy |
| `DOC-REQ-11` | long-form discovery surfaces like `listen`, `blog`, and `projects` should share one reusable content-shell system while keeping per-section identity through theme tokens, sidebar treatment, and mini logos | the site should feel coherent without collapsing distinct sections into one generic page shell |
| `DOC-REQ-12` | each docs section should keep its planning records inside a literal `planning/` or `.planning/` folder instead of mixing planning files with general reference docs at the section root | planning packs, navigation, and agent behavior stay more legible when planning records are physically grouped |
| `DOC-REQ-13` | roadmap, state, task registry, and decisions are the living planning records; one-off implementation plans should be phase-local documents stored under a section planning folder (for example `planning/plans/<phase-id>/...`) | agents should not keep generating free-floating "implementation plan" files that compete with the loop |

## Monorepo scope (canonical narrative)

Narrative for **cross-section** product scope lives **here**, not in repo-root **`REQUIREMENTS.md`** (that file is a **stub pointer**). **`.planning/REQUIREMENTS.xml`** is a machine stub that points to this page.

| Topic | Requirement |
| --- | --- |
| **Books / EPUB** | Optional **planning supplement** trees: `repub epub ... --planning <dir>` and `book.json` **`epubPlanningDirs`** via `@portfolio/repub-builder` and `scripts/build-books.cjs` -- detail in [books planning-docs](/docs/books/planning-docs). |
| **Reader + site** | Books, reader, EPUB pipeline, annotations, and site AI: phased roadmap and discipline in **[books planning-docs](/docs/books/planning-docs)** and **[books task registry](/docs/books/task-registry)**; product boundary and persistence in **[architecture conventions](/docs/documentation/architecture-conventions)**. |
| **Site AI** | Chat should ground on repo-backed RAG, **Sources** panel with snippets and deep links; public chat read-only; repo content is source of truth. |
| **Listen** | **`/listen`** -- unified catalog for **audio tracks** and **BandLab presets**, discovery UX aligned with blog/projects, **group-based shared passwords** for locked rows (no per-user accounts in v1). Planning: [listen planning-docs](/docs/listen/planning-docs). |
| **Shared content surfaces** | `listen`, `blog`, and `projects` should converge on shared discovery/detail primitives, but each surface keeps its own theme language and section mark so users can orient themselves instantly. |
| **Planning workflow** | The living loop should be roadmap-first and foldered: `planning/` contains planning docs, while one-off phase plans live under `planning/plans/` and never replace roadmap/state/task-registry as source of truth. |
