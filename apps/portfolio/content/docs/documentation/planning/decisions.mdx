# Decisions

| Id | Status | Summary | Consequence |
| --- | --- | --- | --- |
| `DOCUMENTATION-DOCS-PLANNING-FIRST` | `accepted` | docs-site planning and requirements should live under the documentation section in the docs tree | documentation improvements should update these records first |
| `DOCUMENTATION-SECTION-OWNS-REQUIREMENTS` | `accepted` | the documentation section itself should carry its own enhancement requirements | docs IA and feature requirements can evolve in public section records |
| `DOCUMENTATION-ROADMAP-VISIBLE` | `accepted` | the docs tree should include a roadmap page that rolls section state up into one readable overview | readers should be able to see what is active, stable, or queued without opening every planning record |
| `DOCUMENTATION-REPO-PLANS-CAN-BE-MIRRORED` | `accepted` | repo-level planning docs such as architecture conventions can be surfaced from the documentation section, but `.planning` stays the source of truth | docs can expose planning context without forking the authoritative records |
| `DOCUMENTATION-MAGICBORN-SECTION` | `accepted` | `magicborn` should exist as a first-class docs section for public worldbuilding material, not as a loose subfolder under books | docs IA reflects the actual worldbuilding domain and gives chat ingestion a clearer source boundary |
| `DOCUMENTATION-SHADCN-THEME-SOURCE` | `accepted` | site chrome uses **shadcn semantic tokens** (`--primary`, `--background`, ...) in `globals.css` with `dark` on `<html>`; portfolio-specific utilities keep working via `@theme inline` aliases (`--color-text` -> `--foreground`, `bg-dark` -> `--background`, etc.) | new UI should prefer shadcn utilities; legacy class names remain valid during migration; headings/marketing may use `text-primary` (brand) vs prose using `--foreground` for hierarchy |
| `DOCUMENTATION-APPS-URL-TOP-LEVEL` | `accepted` | **Interactive tools** (apps hub, embedded repo planner, reader entry from hub) belong under **`/apps/*`**, not under the MDX **`/docs/*`** tree | implement **`documentation-site-06`**: new hub at `/apps`, redirects from `/docs/apps`, link sweep; see [Route conventions](/docs/documentation/route-conventions) |
| `DOCUMENTATION-ROUTES-SINGLE-SOURCE` | `accepted` | URL taxonomy for the public site is authored in **[Route conventions](/docs/documentation/route-conventions)**; task registries reference it instead of duplicating tables | when adding a top-level area, update that page + one registry phase, then implement |
| `DOCUMENTATION-DIALOGUE-FORGE-APPS-URL` | `accepted` | Dialogue Forge **must not** live at **`/dialogue-forge`**; canonical app route is **`/apps/dialogue-forge`** with **308** from the old path | nav, project frontmatter, tests, and docs CTAs must use `/apps/dialogue-forge`; section docs stay under **`/docs/dialogue-forge`** |
| `DOCUMENTATION-PLANNING-ASCII-FIRST` | `accepted` | root `.planning` files and operator-facing planning/reference docs should prefer ASCII punctuation; public-facing prose may keep richer Unicode typography | terminal, CLI, XML, and generated planning surfaces stay readable across tools without downgrading article and narrative presentation |
| `DOCUMENTATION-SHARED-CONTENT-SHELL-THEMED` | `accepted` | `listen`, `blog`, and `projects` should reuse one content-surface component system for discovery and detail pages, but each keeps section-specific identity through theme tokens, mini logos, and local accent treatment | future UX work should extract primitives instead of cloning page shells, while preserving strong "you are here" cues |
| `DOCUMENTATION-SHADCN-REGISTRIES-TRACKED` | `accepted` | external shadcn registries such as `@lucide-animated` and `@magicui` may be tracked as approved sources for future UI passes without being installed immediately | vendoring/docs should record the commands and intended use so future implementation can pull them in intentionally instead of ad hoc |
| `DOCUMENTATION-SECTION-PLANNING-FOLDER` | `accepted` | each section should physically group planning docs under a literal `planning/` or `.planning/` folder instead of scattering them at section root | packs, section navigation, and agent read-order become easier to explain and render |
| `DOCUMENTATION-ROADMAP-FIRST-LOOP` | `accepted` | roadmap, state, task registry, and decisions are the living records; any one-off implementation plan should be phase-local and stored under `planning/plans/`, never as the top-level planning source of truth | planning behavior should stop generating competing `implementation-plan` documents at section root or under `.planning` |
| `DOCUMENTATION-COCKPIT-SINGLE-BODY` | `proposed` | **One** implemented **Repo Planner cockpit modal body** (PlanningCockpit + dashboard), imported by **portfolio** `ModalRoot` and rendered inside **`ReaderModalRoot`** (**`readerModalStore`**) in **`@portfolio/repub-builder`**; **reader planning strip** is **in-package** and opens via **reader** store only (**`BOOKS-READER-PLANNING-STRIP-PACKAGE-READER-MODAL-STORE`**) | finalize after answers in [embed architecture](/docs/documentation/reader-repo-planner-embed-architecture); tasks **`documentation-site-08-*`**, **`books-reader-03-06`**, **`books-reader-03-07`** |
| `DOCUMENTATION-REPUB-REPOPLANNER-MAJOR-PACKAGES` | `accepted` | **`@portfolio/repub-builder`** and **`repo-planner`** are **major publishable** verticals; portfolio is a **reference host** that composes them | publish **`exports`**, **peerDependencies**, and **consumer docs** per **`repo-planner-integration-02`** and **`documentation-site-07`** |
| `DOCUMENTATION-SITE-MEDIA-IMAGE-OPENAI` | `accepted` | **App cover / media copilot** image generation uses the **OpenAI Images API** (same vendor family as chat/embeddings); dedicated route stays separate from RAG **`/api/chat`** | env + server implementation target OpenAI; rate limits and logging follow existing chat patterns |
| `DOCUMENTATION-SITE-MEDIA-COPILOT-CONTEXT` | `accepted` | **Extend** the floating **`SiteCopilot`** for the media workflow — **no manual “media mode” toggle**; launching from a **hover-only control on the card cover** passes **`mediaSlot`** (e.g. `app-cover:reader`) so the session is implicitly scoped to that asset | UI opens copilot with slot context; thread/composer can show session gallery + “set cover” without a separate modal shell unless UX later demands it |
| `DOCUMENTATION-RAG-EMBEDDINGS-LOCAL-MINILM` | `accepted` | **Optional** RAG embeddings use **`RAG_EMBEDDING_PROVIDER=local`** with **`@xenova/transformers`** and **`Xenova/all-MiniLM-L6-v2`** (384-d); weights cache under **`apps/portfolio/models/.transformers-cache/`** (gitignored). **OpenAI embeddings are not required** when this is on. | **Full re-ingest** + **delete `portfolio-rag.db`** (or `RAG_SQLITE_PATH`) when switching dimensions vs OpenAI `text-embedding-3-small` (1536); chat + ingest share the same provider env |
| `DOCUMENTATION-MCP-SHADCN-PNPM` | `accepted` | **shadcn MCP** in **`.cursor/mcp.json`** runs via **`pnpm --dir apps/portfolio exec shadcn mcp`** so the CLI resolves inside **`@portfolio/app`** (avoids **`npx` / npm “Cannot read properties of null (reading 'package')”** when the workspace root has no app `package.json` semantics npm expects) | Cursor must still **enable** the server in Settings; **Supabase** URL MCP may require **separate browser auth** per Supabase MCP docs |
| `DOCUMENTATION-PORTFOLIO-UI-LAYERS` | `accepted` | **`components/ui/`** holds **owned** shadcn-style primitives (edit for design-system needs); **feature composition** lives under **`components/<domain>/`**. Dedupe repeated patterns in **domain** code first; avoid embedding product logic in `ui/`. Canonical prose: root **`AGENTS.md`** (**Portfolio UI**). | Refactors tracked under **`documentation-site-15`**; shared molecules (e.g. **`DiscoveryTagButton`**) live next to consumers in **`components/content/`** unless promoted later |
| `DOCUMENTATION-PUBLIC-OPENAPI-SWAGGER` | `accepted` | **Public** HTTP surface is documented as **OpenAPI 3** (`GET /api/openapi`) from **`lib/openapi/public-portfolio-openapi.ts`**; **Redoc** at **`/docs/api`** loads that spec (standalone page, not the MDX file-tree shell). **Admin**, **owner-only reader writes**, and **media** routes stay off the spec. With **Clerk**, **`middleware.ts`** lists **anonymous** API prefixes (session, content search, artifacts file, planning-pack manifest, docs archive/source, RAG search, listen unlock, OpenAPI) so browsers are not forced through **`auth.protect()`** for those reads. | Extend the spec when adding intentionally public routes; keep **`isPublicRoute`** aligned or anonymous API calls fail in production |
