# Portfolio v2 — requirements (draft)

Living capture for a **full rebuild** of the public portfolio app (today: `apps/portfolio`). Method: same discipline as GAD greenfield work — **stable requirement ids**, **explicit non-goals**, **definition of done per phase**, and **automated gates** where behavior is executable.

**Canonical monorepo narrative today:** [documentation requirements](/docs/documentation/requirements) (product boundaries, auth, data contract, shared surfaces). **Global planning rules:** [global requirements](/docs/global/requirements).

## 1. Intent

| Id | Statement |
| --- | --- |
| `P2-INT-01` | Ship a **maintainable** public site + gated product surfaces without the accumulated complexity of the current Next + Payload + monorepo hoist + tracing workarounds. |
| `P2-INT-02` | Preserve **reader-facing quality** (house copy registers, coherent IA, accessible UI) per root [AGENTS.md](/AGENTS.md) content rules. |
| `P2-INT-03` | Keep **one permission and persistence story** (owner/tenant, entitlements, provider-neutral storage) aligned with **GLOBAL-REQ-09** / **GLOBAL-REQ-10** and documentation [DOC-REQ-16](/docs/documentation/requirements). |
| `P2-INT-04` | Prefer **boring deploys**: predictable serverless bundles, minimal “only on Vercel” failures, explicit env contract. |

## 2. Non-goals (default until promoted)

| Id | Non-goal |
| --- | --- |
| `P2-NG-01` | Pixel-perfect parity with every legacy route on day one. |
| `P2-NG-02` | Re-implement all internal tools (planning pack modal, RepoPlanner cockpit, magicborn CLI UI) inside v2 unless listed in **must carry**. |
| `P2-NG-03` | Public self-signup / multi-tenant SaaS (v1 stays **no public signup**, per existing policy). |

## 5. Technical requirements (candidates)

| Id | Requirement | Notes |
| --- | --- | --- |
| `P2-TEC-01` | **Deploy target** is Vercel (or document if changing). | Drives runtime, env, tracing. |
| `P2-TEC-02` | **Data layer** is explicit: Postgres + ORM/CMS vs “static + API only” vs split (public static / private API). | Affects build-time vs request-time data. |
| `P2-TEC-03` | **No dev-only tools in production routes** (e.g. subprocess `tsx`). | Lesson from auth/session. |
| `P2-TEC-04` | **File tracing / bundle** policy is documented for every server entry that loads DB or S3 clients. | Or: single deploy root like Grime Time to reduce hoist issues. |
| `P2-TEC-05` | **Monorepo**: stay in `custom_portfolio` with `apps/portfolio-v2` **or** extract repo — decision recorded in **DECISIONS** when chosen. | Impacts `pnpm`, CI, submodules. |
| `P2-TEC-06` | **Node** LTS alignment with root `engines` (today ≥ 22). | |
| `P2-TEC-07` | **Secrets**: only env / Vercel; no committed credentials. | |

## 6. Verification & “eval” pattern (GAD-like)

Mirror **gad-27 / gad-28** spirit: comparable runs, scored or pass/fail artifacts.

| Id | Gate |
| --- | --- |
| `P2-VFY-01` | CI: `pnpm install`, `build`, `lint` for the v2 workspace. |
| `P2-VFY-02` | **Smoke routes**: list of URLs (public + one authenticated path) that must return non-5xx after deploy. |
| `P2-VFY-03` | **Data contracts**: any Payload/collection or API schema change has a migration or seed step documented. |
| `P2-VFY-04` | **Optional eval harness**: fixture project(s) that implement a minimal clone of v2 constraints (like game evals for GAD) — only if you want regression compares across refactors. |

## 8. Open questions (fill before phase plans)

2. **Books/reader**: same repo package boundaries (`repub-builder`, `public/books`) or isolate?
3. **Auth**: Clerk + Payload only, or reduce to one identity store?
4. **SEO / rendering**: static generation where possible vs force-dynamic for CMS?
5. **Internationalization** / accessibility targets beyond current baseline?
