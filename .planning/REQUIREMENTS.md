# Books & reader - requirements

Single living document for books tooling and the reader/chat stack. Update this as we flesh out scope.

## Books source of truth

- **Location:** Repo-root `books/<slug>/` (e.g. `books/mordreds_tale/`).
- **Structure:** `chapters/<chapter-dir>/*.md` (or `.mdx`), optional `images/`, optional `book.json`.
- **Metadata:** `book.json` may contain `title`, `author`, `description`. If missing, title is derived from slug.

## Tooling

- **Build script:** Root-level `scripts/build-books.cjs` (invoked via `pnpm run build:books`).
- **Dev:** `pnpm dev` (app workspace) runs a debounced watcher on `books/**` that re-invokes `build-books.cjs` so `manifest.json` and EPUBs stay current without manual rebuilds.
- **Input:** `books/<slug>/` for each slug (directory listing under `books/`).
- **Output:** `apps/portfolio/public/books/<slug>/book.epub` and `apps/portfolio/public/books/manifest.json`.
- **Manifest:** **Autogenerated** aggregate JSON (do not edit by hand). Array of `{ slug, title, description?, hasEpub }`. `lib/books.ts` reads it; see `public/books/README.md`. In **development**, the manifest is re-read on each request (no in-memory cache) so watcher updates show on the homepage.

## Builder package and authoring components

- **EPUB builder package:** The workspace package still lives at `packages/repub-builder`, but its active job in this repo is EPUB generation only. Build books through `pnpm run build:books`; there is no active `.repub` output or release path.
- **book-components:** MDX authoring components (BookRoot, Chapter, Section, ChapterTitle, PartDivider, Callout, Blockquote, Figure, CodeBlock, BookMeta). **Wired:** the EPUB build compiles `.mdx` with `@mdx-js/mdx` and book-components; `.md` uses marked.

## Vendored desktop reader

- **Kookit** (vendor/kookit): [B2Gdevs/kookit](https://github.com/B2Gdevs/kookit).
- **Koodo Reader** (vendor/koodo-reader): [B2Gdevs/koodo-reader](https://github.com/B2Gdevs/koodo-reader). Vendored for desktop reader work; the portfolio itself remains EPUB-only.
- **Embeddable reader:** Portfolio uses the in-app EPUB reader (react-reader / epub.js).

## Format

- **EPUB:** Universal format. Built from markdown/MDX to HTML; images inlined as base64. Served at `/books/<slug>/book.epub` for download. Plain `.md` must not fail for EPUB.

## Compiled book content

- **Today:** Markdown compiled with marked; `.mdx` compiled with `@mdx-js/mdx` and book-components (Callout, ChapterTitle, Figure, etc.) during EPUB build time. Single static HTML document with nav + sections; images in `assets/images/`.
- **Future (optional):** richer author tooling is fine, but the active output target remains EPUB.

## Reader

- **Portfolio reader:** In-app EPUB reader (react-reader / epub.js). Reads `book.epub` at `/books/<slug>/read` and in **BookReaderEmbed** (docs/articles). Location persisted in localStorage by slug.
- **Local EPUB upload:** The books area should allow opening an arbitrary local `.epub` file in the same in-app reader without adding it to the repo.
- **Front-page activation:** The homepage can defer mounting the embedded reader, but the primary CTA must load the featured book into that reader when invoked.
- **Book selection:** The homepage reader area should expose the available books list; entries without a built EPUB should remain visible but disabled as `coming soon`.
- **Koodo Reader (desktop):** Primary consumer-facing desktop reader build (Electron).
- **Standalone:** Same `.epub` file works in any EPUB reader, and the repo should also support a frozen in-browser reader build for local use.
- **Frozen reader builds:** The repo should support versioned non-dev reader builds so the latest frozen build can be run by default while older frozen builds remain launchable by id.

## Reader extensions (annotations and authoring)

- **Annotations:** Highlights and comments anchored with [EPUB CFI](https://idpf.org/epub/linking/cfi/) ranges (via epub.js). Users can list, jump to, edit note text, and delete annotations in the in-app reader.
- **Persistence (tier A):** Portfolio stores annotation sets in **IndexedDB** per book instance, keyed by `storageKey` plus a **content fingerprint** (hash of the EPUB bytes) so a revised rebuild does not silently mix highlights across editions.
- **Export / import (tier A):** Versioned JSON schema (`portfolio-epub-annotations` / `annotations[]` with `id`, `cfiRange`, `quote`, `note`, `color`, `createdAt`, `updatedAt`) for backup and tooling.
- **Embedded payload (tier B):** Optional **"Save annotated copy"** repacks the EPUB zip with `META-INF/portfolio-annotations.json` holding the same schema. Third-party readers ignore unknown files unless they implement this extension; interoperability is best-effort by design.
- **Optional build-time merge:** Future: merge exported annotations into `pnpm run build:books` output for fixed "author editions."
- **Planning in reader:** Optional read-only panel (site-integrated) linking to books planning docs (e.g. Mordred's Tale state/tasks/decisions). **Editing** repo planning from the browser is out of scope for public visitors; gate any future author edit path behind auth or local-only workflows.
- **Reading vs authoring:** Same reader shell; toggles and toolbars expose annotation and export actions without a separate app.

## Site AI assistant (public chat)

- **Purpose:** A lightweight "virtual host" chat for visitors that answers about the site, books, and public creative work in the site owner's voice. **Read-only:** no tools that mutate data in v1.
- **Stack:** CopilotKit (runtime + UI) calling OpenAI on the server. Default model: cost-optimized tier (e.g. `gpt-4o-mini`); configurable via env.
- **Planned RAG backend:** Payload CMS is acceptable here as a slim code-first backend over SQLite for RAG-specific collections such as indexed sources, chunks, ingest runs, and optional message records.
- **Vector storage:** Use [`sqlite-vec`](https://alexgarcia.xyz/sqlite-vec/introduction.html) in the same SQLite environment so embeddings live alongside the RAG store. Its `vec0` virtual tables support KNN search plus metadata / auxiliary columns, but the initial plan is to keep chunk text and citation metadata canonical in Payload and store vectors in `vec0` keyed by `chunkId`.
- **`vec0` shape:** Prefer a narrow `vec0` table with `chunk_id` primary key, `embedding float[1536] distance_metric=cosine`, and only small filterable metadata columns such as `section`, `source_kind`, and `visibility`. Do not use partition keys in v1 unless the corpus grows enough to justify them; the docs warn that oversharding hurts search quality and speed.
- **Embedding model:** Default to OpenAI `text-embedding-3-small` at `1536` dimensions. This is the simplest fit for a modest portfolio corpus: materially cheaper than `text-embedding-3-large`, strong enough for mixed docs/projects/worldbuilding text, and it avoids premature truncation while the corpus is still small.
- **Chunking:** Chat knowledge should come from chunked indexed content, not a hand-curated static "about me" summary. Chunks need source metadata (`title`, `publicUrl`, `sourcePath`, `anchor`, `section`) so the UI can show provenance.
- **Source granularity:** Create one RAG source record per published docs/blog/project page or other individually addressable public page. Do not collapse multiple public pages into one source record family in v1.
- **Chunk storage:** Store the normalized chunk body directly on each chunk record. Do not depend on offset-only reconstruction for v1.
- **FTS scope:** Index chunk text plus selected metadata fields such as page title, headings, aliases, and section labels so BM25 can rescue exact-name and phrase queries.
- **Citation precision:** Preserve section and heading anchor metadata when available so cited chunks can link deeper than the page root.
- **Rebuild scope:** Reindex incrementally by checksum when possible; do not rebuild the entire corpus if only a subset of sources changed.
- **Audit visibility:** Hide soft-deleted sources and old ingest runs from normal admin workflows by default; expose them only in an explicit audit view later.
- **Source-of-truth boundary:** Repo-authored books, docs, blog posts, resumes, and project content remain canonical. The RAG backend ingests and indexes them for retrieval; it does not replace the publishing source.
- **Ingestion scope:** Ingest `content/docs`, `content/projects`, `content/blog`, and books/world-building source material. World-building docs should live under a dedicated `magicborn/` docs section and use short-story-style lore pages. Exclude the actual reader-facing manuscript text by default; prefer character, world, lore, setting, and reference material for book-related chat knowledge.
- **Reindex flow:** Support both a manual/scripted ingest path and an owner-only admin trigger. The script path is the canonical implementation; the admin trigger is convenience only.
- **Reindex visibility:** Chat should only serve the **last fully completed ingest run**. Do not expose partially rebuilt corpora while a new ingest is in progress.
- **Delete handling:** When a source is removed from the repo or excluded from scope, mark the Payload source record as soft-deleted for audit if helpful, but remove its active FTS/vector rows from the retrieval corpus immediately when the new ingest run commits.
- **Identity contract:** Keep **source ids stable** across ingest runs. Chunk ids may be regenerated per run as content changes; citations should anchor to stable source metadata plus the current run's chunk rows.
- **Retrieval contract:** Do not rely on pure vector top-k alone. Use a lightweight rerank/filter pass after vector fetch: metadata visibility checks first, then an **FTS5 BM25** pass over the candidate chunk set before sending the final set to the model.
- **Citation UX:** Use a collapsible **Sources** panel by default; inside it, show chunk snippets/cards with source labels and outbound links to the originating page or file-derived public URL.
- **UX:** Collapsible floating control that expands into a normal chat panel; available on main site surfaces (not only the reader).
- **Conversation storage:** Visitor chat defaults to client-local or session-only history. Do not assume durable server-side transcripts until there is an explicit operational need.
- **Secrets:** `OPENAI_API_KEY` server-only; document in `.env.example`. No keys in client bundles.
- **Admin / ingest boundary:** Public chat and retrieval routes can remain open/read-only, but Payload admin and any ingest/reindex controls are owner tooling protected by **basic auth**.
- **Privacy / abuse:** Rate limiting and logging policy should be tightened in a later hardening pass; v1 assumes low-traffic portfolio use.

## Product boundary and persistence model

- **Product posture:** This repo powers a personal portfolio and creative workspace, not a multi-tenant product or customer platform.
- **Public default:** Visitors should default to read-only reading/browsing plus optional public chat. Public browser sessions must not mutate repo content, manuscript files, or `.planning` records by default.
- **Browser authoring:** If manuscript editing, planning editing, or other creative tooling is exposed in the browser, it must be **local-first** and **export-based** unless an explicit owner-only auth or local-dev gate is added first.
- **Repo-authored content:** Books, blog posts, docs pages, resumes, and curated project/site copy remain file-based in the repo. Do not move that authored content into the database by default.
- **Client persistence:** Reader annotations, chat drafts/history, editor drafts, and UI preferences should live in `localStorage`, IndexedDB, or export files when the workflow is user-facing but not truly account-backed.
- **Server persistence:** The local SQLite setup and likely Vercel deployment environment are not reliable long-term storage for public user work. Anything important to keep must either stay in the repo, be exportable by the user, or be backed by an explicitly chosen hosted store later.
- **Database scope:** Reserve database persistence for narrow owner-critical operational records only (for example messages, lightweight inbox/review state, moderation, indexed retrieval chunks, or other internal records that justify server storage).

## Admin / CMS posture

- **Current admin stance:** `/admin` remains owner-facing and local/dev-first unless a later phase adds explicit auth and hosting requirements.
- **Payload CMS:** Payload is now a valid planned backend choice for the site AI retrieval store and other narrow owner-operated collections. It is **not** the default store for blog posts, docs, books, or other authored site content.
- **Editing gate:** Any future browser path that can write manuscript, planning, or operational records must be clearly gated behind auth, local-only tooling, or both. Do not blur public reader/chat surfaces into authoring surfaces.
- **Planning discipline:** Decide the storage owner first (`repo`, `client-local`, or `server`) before building any new edit/create workflow.

## Verification gates

- **Before any release:** `pnpm install`, `pnpm run build`, and `pnpm run lint` must all pass. Lint is not optional; lint failures block release.

## Portfolio experience redesign

- **Goal:** Reposition the portfolio so the first impression is **author / artist / world-builder**, not **software architect**.
- **Primary entry point:** The landing page should lead with the book and an immediate path to start reading.
- **Secondary creative lane:** Songs and audio work should appear on the home page as a distinct section, using placeholders or BandLab embeds until the final catalog is ready.
- **Resume access:** The site should expose a clear `resumes` destination where tailored resumes can be found quickly and opened in standalone print-friendly pages.
- **Resume source files:** HTML resumes stored in `apps/portfolio/misc/html_resumes/` are the source set for the website resume library; the site should be able to discover them, surface useful metadata from them, and let visitors download the original HTML when needed.
- **Resume design source of truth:** Resume visual direction, print standards, and references should live in `.planning/RESUME_DESIGN_GUIDELINES.md` so new resume variants stay printable and aesthetically consistent.
- **Technical work:** Projects, docs, blog, and tooling still matter, but should live as clear secondary navigation destinations without dominating the first screen.
- **Implementation rule:** Keep the existing Next.js/Vercel stack for the first redesign pass. Match the content sequencing and atmosphere of inspirational sites, not their exact implementation details.
- **Placeholder-friendly:** Fake covers, descriptions, track art, and CTA copy are acceptable in early iterations as long as routes and components work end-to-end.
- **Motion / depth:** The redesign should use layered motion and selective 3D to create presence, but must retain good mobile behavior, reasonable performance, and a reduced-motion fallback.
- **Scrollbar consistency:** Shared shell scrollbars (page, site sidebar, docs sidebar, overlays) should use the same thin visual treatment without causing layout drift when they appear.
- **Design source of truth:** Detailed IA, content order, and 3D decisions live in `.planning/PORTFOLIO_EXPERIENCE_REDESIGN.md`.

## Documentation IA

- **Section-first docs:** The docs area should be organized around active sections such as `books/` and `dialogue-forge/`, not generic buckets like "Core Concepts".
- **Worldbuilding section:** Add a first-class `magicborn/` docs section for public worldbuilding material written in short story form. This section is part of the chat ingestion corpus.
- **Magicborn page schema:** Prefer **canon dossiers** over single-scene lore pages. Each dossier page should group one entity, institution, place, or system into chunk-friendly sections such as an opening vignette, canon registry, known facts, story tensions, and source anchors.
- **Planning-doc oriented:** Each active section should expose planning-style documentation as first-class pages, with `planning docs`, `state`, `task registry`, `errors and attempts`, and `decisions` appearing before general implementation notes.
- **XML-inspired structure:** Those section planning pages should mirror the old XML planning breakdown in page form rather than using vague prose placeholders.
- **Compact schema:** Planning pages should use compact repeated structures that are easy for humans to scan and easy for the site to parse later, favoring fixed fields, tables, and explicit record ids over paragraph-heavy prose.
- **Nested planning folder:** In section navigation, planning pages should live inside a `Planning Docs` folder that is collapsed by default and can be expanded to select those pages.
- **Remove stale docs:** Unsupported or abandoned sections like `richepub` and `book-editor`, plus obsolete top-level docs that only describe those systems, should be removed from the portfolio docs experience.
- **Expandable later:** More section folders can be added as the site grows, but the docs shell should already assume real section folders are the unit of organization.
- **Architecture source of truth:** Repo-level architecture and coding conventions live in `.planning/ARCHITECTURE_CONVENTIONS.md`; section docs can summarize or mirror those rules, but should not silently drift from them.

## Planning packs & homepage modal

- **Agent read order:** Codex/Cursor and human operators should treat **section planning MDX** (`apps/portfolio/content/docs/<section>/planning-docs.mdx` first) as the primary index for section-scoped work, then **`.planning/REQUIREMENTS.md`** and **`IMPLEMENTATION_PLAN.md`**, then root **`AGENTS.md`**. (Encoded in root `AGENTS.md`.)
- **Machine-local hints:** Planning MDX may set YAML frontmatter **`repoPath`** (repo-relative file path) and optional **`taskPhase`** (e.g. `books-ai-01`, `documentation-site-03`) so agents grep/open the correct file despite public URLs living under `/docs/...`. The docs UI only surfaces `title` and `description` in the page header; other frontmatter keys stay off-page.
- **Task identity convention:** Use `<namespace>-<stream>-<phase>-<task>` for section task ids (same shape as [global planning](../apps/portfolio/content/docs/documentation/global-planning.mdx): namespace = docs folder key, stream = product line inside that section). Examples: `books-ai-01-04` (books section, **ai** stream = site chat + planned RAG backend), `books-reader-01-15`, `documentation-site-03-04`, `magicborn-world-01-04`. Phase headings mirror the prefix without the final task segment (e.g. `books-ai-01`). Avoid bare `namespace-phase-task` three-segment ids for new work when a stream name keeps queues distinct.
- **Root plan convention:** `.planning/IMPLEMENTATION_PLAN.md` is a summary and cross-cutting queue, not a second detailed task registry. Section planning pages own section task ids; the root plan should reference those ids instead of re-describing them as a parallel backlog.
- **Demo pack (static):** Ship starter `.md` files under **`public/planning-pack/demo/`** for visitors and template projects. Content is explicitly **example-only**: first phases and requirements must describe the **shape** of a real loop, with a clear banner that the AI or human should **replace** demo text when starting a real project. Include a downloadable **`AGENTS.md`** in the demo folder that reinforces read order and planning-first rules.
- **Real pack (this repo):** **Build-time** export of **section planning MDX** only (pages matching the planning-doc leaf set: `planning-docs`, `state`, `task-registry`, `errors-and-attempts`, `decisions`) into **`public/planning-pack/site/`** plus **`manifest.json`**. **Do not** publish or list repo-root **`.planning/*.md`** in the public gallery; those stay private to the repository.
- **Modal UX:** Open from any registered control (home hero, nav, etc.) via a **modal registry** and **Zustand** app state; two tabs — **Starter template** (demo) vs **This site** (exports); PDF-style thumbnails; hover **download** and **expand**; expanded view renders Markdown in-modal.
- **Per-section `AGENTS.md`:** Optional. The gallery lists **only files that exist**; do not surface placeholder copy for missing section-level agent files.
